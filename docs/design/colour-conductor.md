# Colour conductor — live, reproducible, durable desktop theming

**Status:** Proposed — design note (`docs/design/`). Not yet built. Selects Route 1 (below); reverses ADR-036's *"Noctalia as sole theming authority"* while keeping Noctalia as the shell (#411 / Epic E #427) → ADR-worthy (see §ADR relationship under Rationale). **Reshaped by [ADR-041](../decisions/ADR-041-terminal-authority-tui-theming.md) (2026-07-02):** TUIs now follow the terminal's ANSI palette fleet-wide, so Route 1's "re-add the shared TUI Stylix targets" steps (§Design items 1, and the per-tool halves of 5) are obsolete — the TUI surface follows any terminal repaint for free, and the route reduces to theming the *terminal* (foot) per menu entry. The specialisation/persistence/menu machinery and the GUI-surface plan stand; rework those items against ADR-041 before implementation. **Mechanism revised (2026-07-07):** the specialisation-based switch in items 2/3/6/7 below is superseded by a lighter mechanism — generalise `lib/scheme-pair.nix` (which already resolves Stylix's base16 engine per-polarity outside the normal `stylix.polarity` pipeline, for Ghostty/JankyBorders on Darwin) to an arbitrary Nix-declared menu, render each entry's targets as small per-target derivations (the pattern independently verified in [niedch/nix-omarchy-theme](https://github.com/niedch/nix-omarchy-theme)'s `build-theme.nix`), and switch via a `themes/current`-style symlink. A repo-wide audit of every `config.lib.stylix.colors` consumer found none that needs a specialisation's whole-tree re-evaluation — every Linux-desktop consumer is a pure file/dconf render or resolves via ADR-041's ANSI-role mechanism (the TUI statuslines' conversion landed 2026-07-13 via #411, closing item 5) (see `docs/research/omarchy-theme-switching-validation.md` for the tool-level reload evidence this leans on). The specialisation path stays verified-workable (§De-risk evidence) and is retained as a fallback, not deleted.

## Summary

Make Stylix (via Nix) the single theming authority on the desktop, and let the user live-switch across a **Nix-declared menu of named themes and polarity** — applied to every surface immediately, save that GUI application *chrome* keeps its start-up colours until that app is relaunched on a full scheme change. Noctalia drops from colour authority to a themed-by-Nix shell.

## Motivation

There is no easy way to live (or nearly-live) switch theme and polarity. Three things make it hard today, and rule out the naive fixes:

1. **Split-brain live switching (#411).** Noctalia flips its own surfaces (bar, niri borders, foot's palette) live, but the Stylix-pinned surfaces — the four TUI statuslines (zjstatus, gh-dash, the Claude statusline, macchina), and the rest — only re-theme on a full rebuild.
2. **Not reproducible (#406 / Epic E #427).** The live colour/scheme state lives in Noctalia's GUI-managed `settings.json`, deliberately *not* in git — so the desktop's look cannot be rebuilt from the flake.
3. **Not durable.** Runtime theming is owned by Noctalia v4, frozen into upstream maintenance (v5 is a fresh-install rewrite) — so theming coupled to it breaks at the migration, or if Noctalia is replaced.

**Forces — any solution must satisfy these:**

1. **GUI-app relaunch is acceptable.** It is fine for *some* GUI apps to require a relaunch for a theme *colour* switch to take effect. (Polarity dark↔light still reaches GUI chrome live via the portal; and this boundary is universal — no theming route re-themes a running GTK/Firefox chrome on a full scheme change.)
2. **Declaratively reproducible.** The build is reproducible from the flake; nothing about the look escapes git.
3. **Selection persists across restart.** The theme + polarity the user selected at runtime survives a reboot / re-login — it is *not* reset to the build default on every restart.
4. **The theme catalogue is declared in Nix.** The set of selectable themes is the Nix-declared menu; adding a new theme is a Nix edit + rebuild — no adding themes at runtime.

The **strategic driver** behind the route choice is **durability** — the theming should survive Noctalia v4 and even Noctalia's removal. It is served by making the authority Nix/Stylix, not a shell.

## Design

Stylix (via Nix) is the **single theming authority** on the desktop. The user live-switches across a **Nix-declared menu of named themes (each with its polarity variants)** via a generalised `scheme-pair.nix` + per-target derivations, switched by a symlink swap (not home-manager specialisations — see the mechanism revision above); the selection persists across restart with no login hook. Noctalia is demoted from theming authority to a **themed-by-Nix shell** — it still owns the bar, launcher, notifications, lock, and the rest of the shell surface, but no longer owns colour.

1. **Stylix as single authority — delivered per entry, not via target re-adds.** ADR-041 already scopes this item down from the full TUI/niri Stylix target set to theming the *terminal* (foot) per menu entry (§top-of-file ADR-041 note); the GTK colour override (today Noctalia's `@import` wins the cascade) is the other target-writer surface this item still covers. Under the mechanism revision (§top-of-file, 2026-07-07), neither is delivered by re-adding a `stylix.targets.<x>` writer into the home-manager generation — that write path is a read-only symlink a `current`-swap cannot re-point (`home/nixos/stylix-targets-desktop.nix` removed exactly this class of target for #385) — each is instead rendered per menu entry by the item-2 derivation, with the `current` indirection (item 3) as what foot/GTK actually resolve through. The GTK *settings* and Firefox targets stayed throughout the Noctalia adoption and are untouched here (relaunch-bound, force 1).
2. **A Nix-declared theme menu → one rendered entry, not one specialisation.** Generalise `lib/host-palettes.nix` from one scheme-pair per host to a **named menu** (rose-pine, tokyo-night, gruvbox, …, each with dark/light). Generalise `lib/scheme-pair.nix` — which already calls Stylix's `base16.mkSchemeAttrs` engine directly, per polarity, outside the normal `stylix.polarity` module pipeline, for Ghostty/JankyBorders on Darwin — to resolve every menu entry's base16 hex set the same way. Feed each resolved set through a small `build-theme.nix`-style derivation per target (foot.ini, the GTK CSS, the dconf polarity key, Noctalia's `colors.json`, …), using pure-Nix `builtins.replaceStrings` substitution — no shelling out, no runtime templating engine. A switch copies/symlinks the chosen entry's rendered outputs into place and fires the reload signals (item 7) — non-root, sub-second, no rebuild. *(De-risked: `scheme-pair.nix` is proven in production for two Darwin consumers; `nix-omarchy-theme`'s `build-theme.nix` is a working, shipped implementation of the identical render pattern against a different palette source — see §De-risk evidence.)*
3. **Selection + persistence (force 3), for free.** The active selection is a `~/.local/share/themes/current`-style symlink pointing at one menu entry's rendered output — the *only* live state, a bounded, recoverable delta over the Nix-declared default. No login hook or session-start re-activation is needed: the symlink's target survives reboot on its own, because every app config resolves through the stable `current` path rather than through a home-manager generation. Absent the pointer (fresh machine), the host's Nix-declared default applies (force 2). *(This closes the persistence question the specialisation mechanism needed louis-thevenet/darkman's "persist last selection, re-apply at login" pattern for — mined instead from `nix-omarchy-theme`'s and upstream Omarchy's own stable-path-indirection pattern, per `docs/research/omarchy-theme-switching.md` §3.)*
4. **Invocation — hotkey and/or action menu.** The switch is user-invocable via a **hotkey** (keybind registry, #384) and an **action-menu entry** presenting the named-theme menu (the launcher / action-menu surface, #437 / #442). All resolve to "activate the selected menu entry and persist the pointer."
5. **Statuslines → ANSI-slot references.** Convert the four statuslines (zjstatus, gh-dash, the Claude statusline, macchina) from absolute colour to indexed / ANSI-slot references, so they follow the terminal palette and repaint live. *(Landed via #411, 2026-07-13 — all four converted to ANSI-slot references; gh-dash and zjstatus confirmed feasible against the pinned versions; runtime-verified on metis.)*
6. **Noctalia as a themed-by-Nix shell.** Pin Noctalia's `colors.json` per menu entry's rendered derivation *(de-risked: a separate, module-pinnable file)*; leave its runtime templating/recolour off. Whether Noctalia re-reads `colors.json` on a passive file change, or needs an explicit reload trigger, is unconfirmed — per item 7's rule that no surface's watch behaviour may be assumed, this needs the same explicit-trigger verification as niri/waybar before the swap can be called sufficient on its own (carried to Unresolved questions).
7. **Live-repaint plumbing.** The switch script fires the reload signals so already-open instances repaint: foot OSC (per-pty, not `SIGUSR1` — foot has no config-reload path at all; see `docs/research/omarchy-theme-switching-validation.md` #1), `niri msg action load-config-file`, helix `USR1`, the Noctalia `colors.json` swap, fish universals (the per-surface signal map verified in #143). **Every trigger must be explicit, never passive file-watch:** niri's own auto-watch is inotify-on-file and misses a swapped `/nix/store` symlink target (niri#2658, NixOS-specific), and waybar's `reload_style_on_change` has the same blind spot for a symlinked `style.css` — confirmed in `docs/research/omarchy-theme-switching-validation.md`'s cross-cutting finding. This applies identically regardless of switch mechanism; it is not specific to dropping specialisations.

**What "immediately" means.** On a theme/polarity switch: the terminal palette + the slot-referenced statuslines, shells, niri, and Noctalia's chrome **repaint live**; a **polarity** flip additionally reaches GTK/libadwaita and Firefox web content live via the portal `color-scheme` key; only GUI application *chrome* on a *named-scheme* change keeps its start-up colours until that app is **relaunched** (force 1 — universal to any route).

**How the forces are met.** A single Nix/Stylix authority with every menu entry Nix-built (forces 2, 4); the only live state is the selection symlink, which gives persistence (force 3) by construction — no re-activation hook needed — while staying reproducible: a fresh build lands on the host's declared default. The arrangement survives Noctalia v4 (the durability driver): Noctalia is now just a themed shell, so a v4 → v5 migration only re-points the `colors.json` pin; were Noctalia dropped entirely, the per-tool rendered targets still theme every surface.

## De-risk evidence

- **`scheme-pair.nix` generalisation — green (verified, production precedent), with a shared-schema ripple to plan for.** `lib/scheme-pair.nix` already resolves both polarity halves of a host's base16 scheme by calling `base16.mkSchemeAttrs` directly against `pkgs.base16-schemes`, outside the `stylix.polarity` → `config.lib.stylix.colors` pipeline, for Ghostty's dual-theme config and the JankyBorders hook pair on Darwin. Its core `mkPolarity` resolution step generalises trivially from two hardcoded polarities to an arbitrary Nix-declared menu — the same function, more entries. What isn't trivial: `scheme-pair.nix` resolves through `lib/palette-for.nix` keyed on `hostContext.hostName`, and `lib/host-palettes.nix`'s single-sourced schema (#541, shared with `modules/{nixos,darwin}/stylix-palette.nix`'s "host default" notion) is touched by generalising to a menu, not just `scheme-pair.nix`'s own entry count.
- **Per-target derivation rendering — green (verified against a working implementation), scoped to the render pattern only.** `nix-omarchy-theme`'s `build-theme.nix`/`lib/render.nix` render `{{ key }}`/`{{ key_strip }}`/`{{ key_rgb }}` templates via pure `builtins.replaceStrings` — a shipping, working instance of exactly this design's proposed *render* mechanism, against a `colors.toml` palette source rather than Stylix's. Its separate `url`/`rev`/`hash` theme-fetch schema does not transfer here: this design's menu entries are base16 scheme *names* resolved against the already-nixpkgs-pinned `pkgs.base16-schemes` package (`scheme-pair.nix:27`), not externally git-fetched themes, so no equivalent fetch/pin step is needed.
- **No eval-time-only consumer forces a specialisation — green — verified by repo-wide audit (the statusline conditional closed with #411).** Every `config.lib.stylix.colors` / `config.stylix.{polarity,base16Scheme}` consumer in the Linux desktop wiring was traced (2026-07-07). `home/shared/prompt.nix` is **confirmed** by its own comment to read only the `.ansi` field ("the config-forcing `.hex` is never touched") — an ANSI-role reference, safe under any switch mechanism. The four TUI statuslines (zjstatus, gh-dash, the Claude statusline, macchina) converted to ANSI-slot references via #411 (landed 2026-07-13, runtime-verified on metis) — safe under any switch mechanism, like `prompt.nix`; the harder per-entry hex-render fallback is moot. Unconditionally solid: the portal-color-scheme bridge, every `stylix.targets.*`, and Noctalia's `colors.json` are pure file/dconf renders; niri's own border colour already follows Noctalia's runtime file write today, unrelated to Stylix eval; nothing branches other module logic on polarity or scheme choice.
- **Home-manager-specialisation switch — still green (verified), retained as fallback, not the adopted mechanism.** Stylix's home-manager integration copies the system palette into home-manager via its `copyModules` path as `lib.mkDefault` (read at the flake-pinned Stylix rev, `home-manager-integration.nix`), so a plain specialisation assignment overrides `polarity` + `base16Scheme` with no `mkForce` and no `followSystem = false`, contained to the desktop host's home config. Eval-confirmed against metis's actual config. Kept as the fallback mechanism if the lighter render+symlink approach hits an obstacle during implementation.
- **Noctalia `colors.json` pinning — green (verified).** A separate file from the runtime-mutable `settings.json` (per `docs/desktop/noctalia.md`); the "does Noctalia rewrite it on a flip" concern disappears because Noctalia no longer performs the flip.
- **Still unverified** (carried to Unresolved questions): Noctalia's own explicit reload trigger for `colors.json` (item 6); the exact reload-signal set and its explicit-trigger wiring (niri/waybar symlink-watch blind spot); on-`metis` switch latency and repaint.

## Drawbacks

Reasons one might not do this at all:

- It **reverses the colour-authority half of the Noctalia adoption (#385)** — re-adding the foot/niri/TUI Stylix target-writers and reverting the GTK colour override, undoing work landed only recently.
- It **rebuilds the live-repaint plumbing Noctalia currently provides for free** — the reload-signal fan-out becomes ours to maintain.
- It **gives up Noctalia's in-shell GUI scheme picker** — the switch is driven by hotkey / action-menu instead of a point-and-click panel. (That the live menu is the *declared* set is constraint 4, the desired behaviour — not a drawback.)
- **Registering a new colour-consuming surface is a manual step, not automatic.** Dropping specialisations trades away the module tree's free coverage of any new consumer: adding a future surface that reads the palette means adding it to the per-entry render list, or it silently theme-switches nowhere. This is not new cost relative to today — it is the identical registration `stylix.targets.<x>.enable` already requires, just triggered once per menu entry instead of once per build — but it is a real discipline requirement worth naming rather than assuming away.

## Cost

The accepted standing price, once Route 1 is chosen: **build-time scales with the menu size** — one small per-target derivation set per scheme × polarity via the generalised `scheme-pair.nix`, cheaper to evaluate and build than a home-manager specialisation per entry, since only a handful of text substitutions run rather than re-evaluating a slice of the module tree — plus the manual-registration discipline named in §Drawbacks.

## Rationale & alternatives

Two routes present themselves for *where theming authority lives*:

- **Route 2 — Noctalia stays the conductor; pin a reproducible base via a `settings.json` merge.** Keeps Noctalia's in-shell GUI picker and instant per-surface recolour, and could switch to *any* scheme at runtime without pre-declaring it. **Rejected:** its mechanism *is* Noctalia v4 (the merge targets v4's `settings.json`; v5's model differs), so it fails the durability driver; and it only partially satisfies the reproducibility force.
- **Route 1 — Stylix as single theming authority; live switching across a Nix-declared menu of pre-rendered entries.** Durable (authority is Nix/Stylix, shell-independent) and reproducible by construction. **Selected** — it is the only route that satisfies forces 2–4 and the durability driver. (The switch mechanism within this route was revised 2026-07-07 from home-manager specialisations to per-target derivations + a symlink swap — see the top-of-file note; the authority decision is unchanged.)

**Impact of doing nothing:** the desktop stays split-brain on a switch (#411), non-reproducible (#406), and the theming breaks at the Noctalia v4 → v5 migration.

**ADR relationship.** This reverses ADR-036's "Noctalia as sole theming authority" decision (while keeping Noctalia as the shell), so it warrants an ADR change — an amendment to ADR-036 or a new ADR — landed alongside implementation. This design note owns the mechanism; the ADR owns the authority direction-change.

## Prior art

The live-switching survey in [`../research/prior-art.md`](../research/prior-art.md) §3 surfaced three *mechanism* routes: darkman + home-manager-specialisation `activate` (originally adopted here for Route 1's switch; superseded 2026-07-07 by the lighter derivation+symlink mechanism above — see the top-of-file reshaping note), Quickshell-native IPC (Noctalia's own runtime recolour — the mechanism behind Route 2), and a matugen/pywal cache (ruled out as least suited to the declarative model). The persistence design no longer needs darkman's login-re-activate pattern — the symlink's own persistence made that machinery unnecessary — but darkman remains relevant to §Future possibilities (sunset/sunrise-driven polarity).

**Mined — [`../research/omarchy-theme-switching.md`](../research/omarchy-theme-switching.md) and [`../research/omarchy-theme-switching-validation.md`](../research/omarchy-theme-switching-validation.md).** The foot pty-discovery + OSC-repaint loop, the per-app reload-signal map, and the OSC-`4;N` → ANSI-slot linkage are folded into §Design item 7 and the de-risk/unresolved sections above. The validation note additionally corrected two claims about tools that predate the Noctalia adoption: **fnott has no reload path at all** (not a mako analog — a theme flip loses its live notification queue) and **waybar's "no live reload" was overstated** (SIGUSR2 + `reload_style_on_change` work, with tooltip styles and `-s` paths as the exceptions) — moot for this design specifically, since #385 already retired both from the desktop's active surface set (Noctalia owns notifications and the bar); retained here as corrections to the standing research record, not as claims about item 7's current surface list. A second implementation, [niedch/nix-omarchy-theme](https://github.com/niedch/nix-omarchy-theme), was independently investigated 2026-07-07 and mined for the render mechanism (`builtins.replaceStrings` templating, §De-risk evidence) — its `url`/`rev`(preferred)/`ref`(flagged-impure)/`hash` theme-fetch schema is a template for a *future* externally-sourced-theme extension, not something `lib/host-palettes.nix`'s menu needs today, since its entries resolve against the already-pinned `pkgs.base16-schemes`. **This mechanism revision inverts `omarchy-theme-switching.md` §8's own call:** that note filed the `mv`-swap / `current/` indirection under "diverges — do not adopt," reasoning that the home-manager generation symlink already gave the same property; the 2026-07-07 audit found that belief incorrect for the mechanism this design now specifies (item 3) and adopts the indirection after all — the same class of inversion CLAUDE.md records for ADR-029's waybar/fuzzel/fnott guidance.

## Unresolved questions

To close during implementation:

- The generalised `scheme-pair.nix` + per-target derivation shape: exact target list per menu entry, and where the render functions live (co-located with each `stylix.targets`-equivalent, or centralised); the `host-palettes.nix`/`palette-for.nix` schema ripple from generalising a per-host default to a menu (#541).
- The switch-script's exact trigger list, driven explicitly per `docs/research/omarchy-theme-switching-validation.md`'s cross-cutting finding (niri/waybar symlink-watch blind spot) — no passive file-watch may be assumed for any surface, **including Noctalia's `colors.json`** (item 6 currently assumes a passive re-read with no named trigger).
- On-`metis` console verification: switch latency, statusline + Noctalia repaint, that the `themes/current` symlink genuinely survives reboot with every app config resolving through it, and that no two-writer incoherence appears during the transition.

## Future possibilities

The design enables but does not commit to these:

- **Automatic polarity by sunset/sunrise** via `darkman` driving the polarity flip on a schedule, reusing the same switch/persist path (darkman triggers the symlink swap instead of a specialisation activation).
- **Per-app theme overrides** beyond the single host default — a surface electing a different menu entry.
