# macOS within-Space tiling — deterministic window placement that leaves Spaces to macOS

**Status:** Proposed — design note (`docs/design/`). Not yet built. Selects **Route A (adopt Amethyst)** over Route B (build a Hammerspoon within-Space tiler); both weighed below. Adoption is gated on an on-box de-risk trial of the Electron app set (§De-risk, §Unresolved) — the route is chosen, the trial confirms it ships. Amends [ADR-039 §7](../decisions/ADR-039-capability-registry.md) — adds Amethyst to a considered set §7 never evaluated, and augments §7's "no tiling WM, stateless geometry hotkeys" with automatic within-Space tiling while keeping the native-fullscreen-Spaces model unchanged → ADR-worthy (#440).

## Summary

On neptune, give the *non-fullscreen* case deterministic, automatic tiling: new windows fill the current native Space's visible area by a fixed layout rule (the Hyprland-style "windows open into a tiled layout" feel), instead of opening wherever macOS drops them and being arranged by hand. The tiler operates **only within the current Space** and never owns workspaces — macOS native Spaces stay the workspace mechanism (the [ADR-039 §7](../decisions/ADR-039-capability-registry.md) model), and a window thrown to its own full-screen Space simply falls out of the tiled set. The recommended realization is **adopting Amethyst** (a maintained, no-SIP, Accessibility-only tiler that already works exactly this way and is configured from a git-managed YAML file), in preference to building a bespoke tiler in Hammerspoon.

## Motivation

[ADR-039 §7](../decisions/ADR-039-capability-registry.md) froze the macOS interaction model as **pure Hammerspoon, no tiling WM**: maximized "columns" are native full-screen Spaces, and *within* a Space windows are placed by **stateless geometry hotkeys** (`Hyper+C` center, `Hyper+R` width-cycle, `Hyper+M` maximize, `Hyper+±` grow/shrink — see [macos-window-management.md](../desktop/macos-window-management.md)). That settled the *workspace* axis well — native Spaces are stable and supported — but it left the *within-a-Space* axis manual: a new window opens at whatever position and size macOS chooses, and the operator hand-arranges it chord by chord. There is no determinism of *open*: launch two terminals and a browser and you get an overlapping pile, not a layout.

The objective is the Hyprland pattern: when a window opens on a non-fullscreen Space, it should **automatically** take a predictable share of the visible area (dwindle/BSP-style split of the focused region), so a working set self-arranges. This is the one piece of niri/Hyprland behaviour §7 consciously did *not* deliver — and §7 declined it because, at the time, every tiler it surveyed was disqualified: AeroSpace (wrong paradigm), Paneru/OmniWM (0.x single-maintainer), PaperWM.spoon/custom tiler (stateful fragility), yabai (SIP), Magnet (non-declarative). Notably that list **never evaluated Amethyst** — the one mature, maintained, within-Space, no-SIP, declaratively-configurable BSP tiler. That gap is the opening this note works.

**Forces — any solution must satisfy these:**

1. **Within-Space only.** Tiles the visible area of the *current* native Space; it must not enumerate, create, or move windows across Spaces. (For a Hammerspoon build this means never calling `hs.spaces` — the private-API surface Apple already broke on Sequoia, Hammerspoon #3698; for a standalone tiler it means the tiling loop is per-Space and Space-switching is deferred to macOS.)
2. **Complementary, not a workspace layer.** Native Spaces remain the workspace mechanism (§7 preserved). The tiler must not impose its own workspaces. A window thrown to its own full-screen Space (`Hyper+F`) must fall out of the tiled set with no coordination.
3. **Deterministic open.** A newly-opened window lands in a predictable position by a fixed layout rule — not wherever macOS drops it.
4. **No SIP.** Accessibility-API only; the repo's no-SIP-disablement posture is non-negotiable (this is why yabai was rejected in §7).
5. **Declaratively configured.** Layout, rules, and keybinds live in git; no GUI-only state that escapes the flake (explicit > implicit).
6. **Survives the Electron-heavy app set.** Chrome, Cursor, Claude, ChatGPT, Slack, Obsidian must tile correctly — which means the `AXEnhancedUserInterface` resize-correctness dance must be handled, and custom-chrome windows classified by **bundle ID**, not titlebar presence (the PaperWM.spoon #159 lesson).
7. **Robust / self-healing — the anti-fragility force.** §7 rejected a "custom tiler" *specifically* for stateful-tiler fragility. Any tiler adopted here must not reintroduce that: layout state must be a cache recomputable from the live window list, never a persisted source of truth that can desync.
8. **Keybind single-sourcing.** [ADR-039](../decisions/ADR-039-capability-registry.md) makes `lib/capabilities.nix` the one source for binds, respecting the Karabiner-reserved `Ctrl+Opt` chords (`Ctrl+Opt+arrows` Mission-Control, `Ctrl+Opt+1‑9` Space jumps). A solution should flow binds through that registry, or — where it structurally can't — carry a documented, lint-checked exception rather than a silent second source.

## Design

**Route A — adopt Amethyst (recommended).** Amethyst is a standalone Accessibility-API tiling WM that already embodies forces 1–4 by construction: it tiles *within* each native Space, tracks layout per-Space/per-screen, and **defers Space switching to macOS** (it has no internal workspace layer; it provides binds to *throw* windows onto native Spaces rather than invent its own). It ships a **Binary Space Partitioning** layout (alongside Tall/Wide/Columns/…) and reflows automatically as windows open and close — the deterministic-open behaviour the Motivation asks for. It requires only the Accessibility permission, never SIP.

The mechanism on neptune:

1. **Install the app via Homebrew cask `amethyst`** (it is not in nixpkgs), declared in `modules/darwin/homebrew.nix` — the same clause-1, no-nixpkgs-Darwin, cask-with-Sparkle shape already used for Hammerspoon and Ghostty ([ADR-031](../decisions/ADR-031-nix-homebrew-boundary.md)).
2. **Configure declaratively** via `~/.config/amethyst/amethyst.yml`, written by a new `home/darwin/amethyst.nix` `home.file`/`xdg.configFile` entry. The YAML config **overrides the GUI prefs** (verified, §De-risk), so nothing about the layout escapes git. Set: `layouts: [bsp, tall, floating]`; `mod` keys and `mod1`/`mod2` bindings chosen to sit **off** the Karabiner-reserved `Ctrl+Opt` chords; `window-margins`/gaps to taste. The `floating` bundle-id rules are *not* pre-seeded on a forecast — the trial (§Unresolved) determines which of the Electron apps actually misbehave, and only those get floated.
3. **Complementarity with §7 is automatic** (force 2): Amethyst tiles whatever windows are on the current Space. `Hyper+F`'s native-fullscreen window lives on its own Space, so it is simply not in the current Space's tiled set — the two models compose with zero coordination. Native Space navigation (`Ctrl+←/→`, `Ctrl+1‑9`) is untouched; Amethyst's own throw-to-Space binds are left **unbound**, so macOS stays the sole workspace driver.
4. **Electron correctness (force 6)** is upstream's responsibility — Amethyst, not us, owns the `AXEnhancedUserInterface` resize handling. This is the one assumption that must be trial-verified on neptune's actual app set before the route is committed (§De-risk, §Unresolved).
5. **Keybinds (force 8)** are Amethyst's decisive cost: its bind set lives in its own YAML, *not* emittable from `lib/capabilities.nix`, and ADR-039 §8's collision lint reasons only over *registry-emitted* binds — so it does **not** reach the Amethyst YAML as things stand. The reconciliation is to (a) choose binds that never collide with the Karabiner reservations, and (b) record them as an exception in [keybinds.md](../desktop/keybinds.md). Whether the collision lint can be *extended* to read the Amethyst config is an open question, not a settled mitigation (§Unresolved) — until it is, the exception rests on a doc convention, the weakest rung of ADR-032's enforcement ladder, guarding a correctness-severity concern. This is a deliberate re-introduction of a second, un-emitted keybind surface — the founding drift ADR-039 exists to kill, knowingly re-admitted for one tool, and weighed in Cost.

**Route B — build a within-Space BSP tiler in Hammerspoon (recorded, not recommended).** Stay in the existing tool: subscribe to `hs.window.filter` create/destroy/focus events on the *current screen only*; maintain an in-memory BSP tree; reflow into `hs.screen:frame()` (visible area, Dock/menu-bar respected); never call `hs.spaces` (force 1 by construction). The anti-fragility force (7) is met by making the tree a **cache, not the source of truth** — a single `retile()` rebuilds the layout from the live `hs.window.filter:getWindows()` enumeration, so any desync self-heals on one keypress. Binds emit natively from the capability registry (force 8 — its sole advantage). The cost is decisive against it: it is net-new custom Lua with **no prior art to fork** (verified — every community Hammerspoon auto-tiler touches `hs.spaces`; Hammerspoon ships no production BSP tiler), it makes the `AXEnhancedUserInterface` dance *our* problem to implement and maintain (PaperWM.spoon omits it, which is *why* its Electron handling is rough), and it is precisely the "custom tiler" §7 rejected for stateful-tiler fragility — now owned by a solo operator on a daily-driver box.

**How the forces are met (Route A).** Within-Space-only, complementary, deterministic-open, and no-SIP (forces 1–4) hold by Amethyst's design, as attested by its docs (not yet on-box; §De-risk). Declarative (5) holds via the YAML-overrides-GUI config in git. Electron correctness (6) is delegated upstream — the strongest available answer, pending on-box trial. The two forces that genuinely pull against each other are **anti-fragility (7)** and **keybind single-sourcing (8)**, and Route A trades one for the other: it wins 7 (a maintained, widely-used tiler, with Electron correctness owned upstream, instead of bespoke Lua we maintain) and *loses* 8 — it breaches the single-sourcing guarantee that is ADR-039's whole reason to exist. That breach is not waved away by precedent (§7 never adjudicated 7-vs-8 — rejecting the custom tiler happened to serve both at once); it is accepted on its own merits, argued in Rationale: the breach is *bounded* (a small static bind set, one tool) where the bespoke-maintenance burden it avoids is *open-ended*. The recommendation is therefore contingent — Route A stands only if the de-risk trial shows Amethyst actually handles the Electron set; if it does not, that upstream-correctness win evaporates and the calculus reopens.

## De-risk evidence

Verification to date is **documentary, not on-box** — upstream README/config-docs/Homebrew and primary source for the AX technique, read 2026-06-27. Nothing here has been checked against neptune's running system or the flake pins (unlike, say, colour-conductor's eval-confirmed-against-metis de-risk); the tags below say "attested" where the only evidence is the vendor's own docs, and "verified" only where a check beyond vendor claims was run. The one make-or-break assumption is explicitly *not* retired and is carried to Unresolved.

- **Tiles within native Spaces, no own-workspace layer — attested (documentary).** Amethyst README + `.amethyst.sample.yml` (`development` branch, read 2026-06-27): per-Space/per-screen layout, defers Space switching to macOS, ships `throw-space-*`/`follow-space-thrown-windows` binds (throw-to-native-Space, not own workspaces). Vendor's own claim; not yet observed on-box.
- **No SIP — attested (documentary), corroborated.** README requires Accessibility permission only; no `csrutil`/SIP mention (absence of evidence). Corroborated by the yabai SIP-feature split: BSP tiling + in-Space resize need Accessibility only; SIP is required solely for Space create/destroy, move-to-Space, sticky, opacity, layers, animations, PiP — none used here.
- **BSP layout + file-based config that overrides the GUI — attested (documentary).** README lists BSP among 14 layouts; `docs/configuration-files.md` states search order `~/.amethyst.yml` → `~/.config/amethyst/amethyst.yml` and that a present file **overrides** GUI settings — home-manager-friendly. Non-declarable residue: the one-time Accessibility TCC grant (runbook, same as Hammerspoon).
- **Maintained + installable — verified (checked beyond vendor claims).** Latest release v0.24.3 (2026-04-02), commits through April 2026 (GitHub API); Homebrew cask `amethyst` at v0.24.3. Not in nixpkgs (confirmed absent) → cask path, ADR-031 clause 1. These are API/registry checks, not README assertions.
- **`AXEnhancedUserInterface` resize technique is real — verified for the pattern; Amethyst's *own* handling unverified.** The disable-before-`setFrame`/restore-after pattern for Chromium/Electron is confirmed in source for yabai (`AX_ENHANCED_UI_WORKAROUND`) and Rectangle (`AccessibilityElement.swift`), grounded in Chromium's accessibility design doc. **Unverified:** whether Amethyst implements it, and whether it works on neptune's specific set (Chrome, Cursor, Claude, ChatGPT, Slack, Obsidian).
- **Still unverified (load-bearing, carried to Unresolved):** (a) Electron-set tiling correctness on neptune — verifiable only by an on-box trial, not from Linux/CI; (b) that Amethyst's BSP reflow *feels* like the Hyprland dwindle pattern the operator wants, not merely xmonad-style reflow; (c) the exact keybind mapping that stays clear of the Karabiner reservations.

## Drawbacks

Reasons one might not do this at all (the standing price of the keybind surface is a *Cost*, not a reason to abandon the direction — it lives in §Cost, not here):

- **§7 deliberately chose *no* tiling WM at all.** Maybe stateless geometry hotkeys + native fullscreen are genuinely enough day-to-day, and automatic tiling is solving a problem the operator doesn't really have. The honest test of this is the trial — if BSP-on-open isn't a daily win, doing nothing is the right answer. (The strongest reason-against.)
- **It introduces a second window-management *tool*.** Even setting aside keybinds, a standalone WM running alongside Hammerspoon is more moving parts on a daily driver — another agent holding Accessibility, another thing that can wedge — than extending the tool already in place.
- **More cask GUI-app surface.** Another Sparkle-updating cask to track, on a box that already leans on the cask path for several apps.

## Cost

The accepted standing price once Route A is chosen: **a second keybind surface outside the single-source registry** — the Amethyst YAML, reconciled by hand against the Karabiner reservations and [keybinds.md](../desktop/keybinds.md). The hazard is *drift*, not edit-frequency: precisely because the set is small and rarely touched, a collision introduced elsewhere (a new Karabiner reservation, a registry bind that grows onto a chord Amethyst already uses) rots **unnoticed** — the un-emitted surface is exactly what ADR-039's single-sourcing exists to prevent, and a rarely-audited manual sync point is arguably a worse instance of it than a frequently-edited one. The cost is bounded in *size* (one tool, a static bind set), not in *risk*, until/unless the collision lint is extended to read the Amethyst config (§Unresolved) — which would convert this from a doc-convention to a mechanical guard and substantially retire the cost.

## Rationale & alternatives

The operator's initial lean (the discussion that led here) was toward **Route B** — keep window management inside Hammerspoon, the tool ADR-039 already single-sources. That instinct is well-founded on force 8, and the recommendation below overrides it only because the *maintenance* axis cuts the other way; the override is the thing this section has to earn, not assume.

The decision turns on a genuine trade between two forces that pull opposite ways — **force 7 (anti-fragility / maintenance) and force 8 (keybind single-sourcing)** — since forces 1–6 are satisfiable by *either* route (Route A by upstream design; Route B by careful construction):

- **Route B (Hammerspoon custom tiler)** wins force 8 cleanly — binds emit natively from the registry, no second surface. On force 7 it is *better than §7's blanket "custom tiler = fragile" verdict implies*: the cache-not-truth design (state recomputed from the live window list, never persisted) is a real, direct answer to stateful-tiler fragility, and the within-Space scope keeps it off `hs.spaces` entirely. So force 7 is not "lost hard." What Route B cannot answer is the *maintenance* half of force 7: it is net-new Lua with no prior art to fork, and it makes the `AXEnhancedUserInterface` Electron dance **our** code to write, debug across macOS releases, and own forever — with no upstream. That open-ended burden, on a solo-operator daily driver, is the reason it loses — not a fragility strawman.
- **Route A (Amethyst)** loses force 8 (the un-emitted second keybind surface — a real breach of ADR-039's founding guarantee, §Cost) but wins the maintenance half of force 7 decisively: a mature, widely-used tiler maintains the fragile parts, and Electron correctness is upstream's problem, not ours. **Selected.** The bounded, one-tool keybind breach is judged a smaller standing cost than owning a bespoke tiler indefinitely. Adoption is gated on the de-risk trial confirming Amethyst handles the Electron set; were that to fail outright, the maintenance win evaporates and the choice would reopen (likely toward "do nothing", since Route B's burden isn't worth paying for a tiler we'd then also have to fix) — but the *direction* is settled.
- **Do nothing (§7 status quo — no tiling, stateless hotkeys).** The fallback if the trial shows automatic tiling isn't worth a second tool, *or* if Amethyst fumbles the Electron set. Impact of doing nothing: within-Space placement stays manual; the determinism-of-open the operator asked for is simply absent.

**Already rejected in [ADR-039 §7](../decisions/ADR-039-capability-registry.md)** (not re-litigated here): AeroSpace (i3 tree, not the right paradigm; owns its own workspaces), Paneru/OmniWM (0.x single-maintainer), PaperWM.spoon (leans on `hs.spaces`; *supersedes* rather than complements the native-fullscreen model — the operator explicitly rejected that approach), yabai (SIP), Magnet (non-declarative). The material change this note introduces is **Amethyst**, which §7 never evaluated and which clears every reason the others were rejected.

## Prior art

- **Amethyst (ianyh/Amethyst)** — the model *and* the recommended tool: within-native-Space BSP tiling, no SIP, file-based config, actively maintained. The direct precedent that the desired architecture is sound and already exists.
- **yabai** — confirms the Accessibility-only envelope (the SIP-feature split) and the canonical `AXEnhancedUserInterface` resize workaround (`AX_ENHANCED_UI_WORKAROUND`), mirrored by **Rectangle** (`AccessibilityElement.swift`) and grounded in Chromium's accessibility design doc.
- **PaperWM.spoon (mogenson/PaperWM.spoon)** — cautionary: the only Hammerspoon-native auto-tiler, but it rides `hs.spaces` (broken on Sequoia for cross-Space moves, Hammerspoon #3698), clamps off-screen columns to slivers, and lacks the `AXEnhancedUserInterface` dance (→ its Electron friction, #159). It *supersedes* the native-Spaces model rather than complementing it — the reason it was set aside before this note.
- **This repo** — [ADR-039 §7](../decisions/ADR-039-capability-registry.md) (the frozen stance amended here) and [macos-window-management.md](../desktop/macos-window-management.md) (the §7 selection record). The full verification survey behind this note is suggested as a `docs/research/` companion (see Unresolved) rather than restated here.

## Unresolved questions

- **Electron-set tiling correctness on neptune (de-risk before committing the route).** Trial Amethyst on the box and confirm Chrome, Cursor, Claude, ChatGPT, Slack, Obsidian tile/resize correctly, seeding bundle-id float rules for any that don't. Resolvable only on-box, not in CI.
- **Does BSP-on-open feel like Hyprland dwindle?** Confirm the reflow behaviour matches the operator's intent during the trial, vs falling back to a different Amethyst layout.
- **Keybind reconciliation (implementation).** The exact Amethyst bind set clear of the Karabiner `Ctrl+Opt` reservations, and how far the existing collision lint can reach a non-registry config — full exception vs partial integration.
- **Keep or retire §7's stateless geometry hotkeys?** Whether `Hyper+C/R/M/±` stay as a floating-window fallback alongside Amethyst, or are retired now that auto-tiling covers the common case.
- **Capture the verification survey as `docs/research/macos-tiling-prior-art.md`?** The Amethyst/yabai/PaperWM/Hammerspoon-primitives findings are substantial and load-bearing; convention favours a research note this design cites. Suggested, not assumed — pending endorsement (scope discipline).
- **Out of scope:** multi-monitor tiling (neptune is single-display today — defer until it grows a display; Amethyst's multi-monitor is its weakest area); cross-Space moves (native `Ctrl+←/→` stays the mechanism); drag-to-reflow fidelity beyond what Amethyst provides.

## Future possibilities

- **A thicker `home/darwin/amethyst.nix` module** that generates the YAML from Nix values (e.g. sharing the float-list with other per-app config) rather than a static `home.file`.
- **Amethyst's beta JS Custom layout** for a more niri/Hyprland-exact column behaviour, if stock BSP proves close-but-not-quite.
- **Route B kept on the shelf.** If Amethyst proves insufficient (Electron friction, the keybind-surface cost grating), the Hammerspoon within-Space tiler design above is the recorded fallback — its cache-not-truth principle is the part worth keeping.
