# Claude Code session continuity — Syncthing mesh over `projects/`, macOS-canonical paths, hub archive

**Status:** Proposed — design note (`docs/design/`). Not built. Issue: [#530](https://github.com/dannyfaris/nix-config/issues/530). On acceptance this amends two standing positions via a new ADR: the ADR-034 no-backup posture (the hub archive is backup tooling, and ADR-034's own migration trigger — irreplaceable local-only data accumulating — is precisely this design's premise) and the CLAUDE.md "agent memory is per-host and never synced" stance (rewritten, see §Design).

## Summary

Sync `$HOME/.claude/projects/` — Claude Code's session transcripts and per-project agent memory — across the four personal-side hosts (metis, nixos-vm, neptune, saturn; mercury, the work-only box, is deliberately excluded) with a narrowly-scoped Syncthing mesh, so any local CLI session can be resumed on whichever machine the operator is at, with full conversation history. Three supporting mechanisms make this correct rather than merely plumbed: every participating Linux host bind-mounts `/home/dbf` at `/Users/dbf` and launches `claude` from the `/Users` namespace, so session keys (which encode the absolute cwd) are byte-identical across macOS and Linux; the 30-day default retention window stays, with `cleanupPeriodDays` pinned explicitly so upstream default changes cannot silently alter fleet behaviour; and metis (always-on Linux hub) runs an append-only archive job that copies `projects/` to a location outside both the synced folder and Claude's config dir, giving long-term transcript storage that neither the retention reaper nor a propagated deletion can ever touch. Transport is tailnet/LAN-only — global discovery and relays off. No third-party service, no hosted repo, no dependency on the originating machine being awake.

## Motivation

The operator works across several machines in a normal week. A Claude Code session started on one host is invisible on every other: transcripts live in per-host `~/.claude/projects/`, resume is keyed to the local filesystem path, and nothing syncs. Continuing work on another machine means rebuilding context from CLAUDE.md and git — an approximation, not the conversation. Anthropic's native cross-device features do not cover this: cloud sessions teleport one way (web → local), Remote Control requires the origin machine alive and the session running, and CLI-to-CLI cross-machine resume is a long-standing open feature request (anthropics/claude-code #31992, #44063, #52052) with no commitment. Waiting is not a plan; designing for easy retirement if it ships is (§Future possibilities).

Two secondary needs fall out of the primary one. First, sessions must be recoverable beyond Claude Code's default 30-day local retention — the operator wants long-term transcript storage, distinct from the live resume window. Second, transcripts are plaintext and can contain source, credentials, and client context, so replication must stay strictly within infrastructure the operator controls.

The forces any solution must satisfy:

- **F1 — Full-fidelity roaming.** A session started on host A appears in `claude --resume` on host B with complete history, without the originating machine being awake.
- **F2 — No third-party medium.** No hosted git repo, no sync SaaS; peer-to-peer within the operator's own fleet and tailnet.
- **F3 — Declarative.** The entire fleet is Nix-managed (NixOS + nix-darwin + home-manager); the mechanism must be expressible in the repo, per-host manual steps limited to documented one-time bootstrap.
- **F4 — Survives the reaper, deliberately.** Claude Code hard-unlinks transcripts older than `cleanupPeriodDays` (default 30) on a startup-adjacent schedule, and a bidirectional mesh faithfully propagates those unlinks. Long-term storage must be structurally immune to this, not configured around it.
- **F5 — Identical session keys across platforms.** Transcripts are keyed by encoded absolute cwd (`/Users/dbf/x` → `-Users-dbf-x`). macOS homes live at `/Users/dbf`, Linux at `/home/dbf`; unless unified, synced transcripts are invisible to `--resume` cross-platform.
- **F6 — Sensitive-data posture.** Plaintext transcripts never transit third-party infrastructure; transport is tailnet/LAN, encrypted device-to-device.
- **F7 — Migratable roles, easy decommission.** The hub role must be a flag, not a hardcoded host (the operator expects it may move off metis later), and the whole system must be removable if native sync ships.

## Design

### Mesh: one Syncthing folder, scoped to `projects/` only

A single Syncthing folder, id `claude-projects`, path `~/.claude/projects` on every participating host — never the whole `~/.claude`, which holds credentials, OAuth state, and machine-local caches. Four hosts (metis, nixos-vm, neptune, saturn) share it as `sendreceive` peers. mercury is deliberately excluded: joining would put personal transcripts at rest on a work AWS box and fan work-context transcripts to every personal machine, and the operator chose the clean boundary over roaming work sessions — the cost of that exclusion is recorded in §Drawbacks.

Deployment is platform-split, mirroring the Tailscale precedent (NixOS module on Linux, different mechanism on Darwin): the Linux hosts use the NixOS system module `services.syncthing` (run as `user = dbf`, data under `/home/dbf`) — a system service needs no login session, which the always-on hub requires anyway, and it is the only module with declarative `ignorePatterns` (verified at the pinned nixpkgs rev); the Darwin hosts use the home-manager `services.syncthing` module, which runs on Darwin (verified: the core service has no platform assertion, only the tray does) with the same declarative devices/folders/options surface. Device IDs are public keys — they are pinned in the repo exactly as the fleet's SSH host keys are (declarative device graph; `overrideDevices`/`overrideFolders` on, so stray devices/folders introduced via the GUI are reverted). Bootstrap per node is one-time: first activation generates the keypair, the operator records the printed device ID into the module's device map and re-switches the fleet.

Transport posture (F2/F6): `relaysEnabled = false`, `globalAnnounceEnabled = false`, `localAnnounceEnabled = true`, with device addresses pinned to tailnet names. Relay traffic would be E2E-encrypted regardless, but tailnet-only is the cleaner stance for client-work transcripts, and Tailscale is already fleet-wide. This couples the mesh to Tailscale (interim infrastructure — the pre-Twingate posture is recorded in `docs/runbooks/headless-bootstrap.md`); if Twingate replaces it, device addresses are the one knob to revisit.

Ignore patterns: the Linux hosts declare `.DS_Store` via the NixOS module's `ignorePatterns`, blocking macOS noise from entering the Linux nodes; the Darwin modules lack the option, and mac↔mac exchange of `.DS_Store` is harmless (whether to add an activation-copied `.stignore` on Darwin is an implementation-stage question). `.sync-conflict` files are deliberately *not* ignored — they are the conflict-preservation mechanism and must roam so whichever machine the operator is at can resolve them.

A stances-style eval check (`lib/stances.nix` pattern, ADR-033) asserts every declared Syncthing folder path across the fleet — the NixOS option tree on Linux, the home-manager one on Darwin — ends in `.claude/projects`: the "nothing outside `projects/` is ever synced" guarantee held at eval time, so a future folder addition that widens scope fails `nix flake check` loudly. Guarding "credentials never sync" is security-severity, so a mechanical gate is proportionate under ADR-032.

### Path unification: macOS is the canonical namespace, Linux bind-mounts into it

Every participating Linux host (metis, nixos-vm) mounts `/home/dbf` at `/Users/dbf` (`fileSystems."/Users/dbf" = { device = "/home/dbf"; fsType = "none"; options = [ "bind" ]; }` plus a tmpfiles rule creating `/Users/dbf` as the mount point), and `claude` is always launched from `/Users/dbf/...` paths on Linux. Symlinks cannot do this job: Claude Code calls `fs.realpathSync()` on startup and stores the canonical path (verified — upstream issues #39594/#44456 track it as a complaint), so a `/Users/dbf` symlink would still key sessions under `-home-dbf-...`. Bind mounts are real VFS mounts and survive canonicalisation. The direction is forced: macOS has no native bind mounts (`/home` is claimed by autofs; the reverse mapping needs macFUSE), while Linux-mimics-macOS is two declarative lines per host.

Ergonomics: a fish wrapper on the participating Linux hosts (`home/nixos/`, gated with the bind mount) intercepts `claude` launched from a `/home/dbf/...` cwd and re-execs it from the `/Users/dbf/...` equivalent — deterministic remap, not a warning the operator can miss. Cosmetic cost: inside sessions on Linux, tools report the cwd as `/Users/dbf/...`; accepted. Non-interactive invocations that bypass fish (scripts calling `claude -p` from `/home` paths) key under the old namespace — a cosmetic inconsistency for print-mode sessions, documented and accepted rather than chased.

Cutover: the day the wrapper lands, a Linux host's *existing* sessions and per-project memory live under `-home-dbf-*` encoded directories, which the `/Users`-keyed picker and memory lookup would no longer find — and which the reaper would then silently clear over the following 30 days. Each Linux host therefore performs a one-time migration before joining the mesh: rename every `projects/-home-dbf-<rest>` directory to `projects/-Users-dbf-<rest>` (a rename preserves file mtimes, so no retention clock is disturbed). Related one-time friction, accepted: `~/.claude.json` project trust/config is keyed by absolute path and stays per-host, so each project prompts for trust once under the new namespace.

### Retention: 30-day live window, pinned explicitly

The fleet keeps the effective default: `cleanupPeriodDays: 30`, but pinned explicitly in every host's `~/.claude/settings.json` via the existing ADR-024 jq-merge activation pattern (Claude Code chokes on Nix-store-symlinked settings — upstream #3575 — and the file is runtime-mutated, so merge-don't-clobber is established practice). Pinning matters because the design's correctness window — what "live everywhere" means, what the archive exists to catch — is derived from this value; an upstream default change must not silently alter fleet behaviour. Never `0`: historically it silently disabled transcript persistence entirely (#23710); in the pinned 2.1.191 the settings schema rejects non-positive values outright and the sweep code special-cases `0` to "skip" — both verified in the shipped binary — but the footgun's history alone disqualifies it.

Because retention is uniform at 30 days fleet-wide, the mesh has no sweep-ordering hazard: any transcript one node would reap is already reap-eligible everywhere, and the deletion the mesh propagates is the policy, not an accident. The archive (next section) is what makes the policy safe.

### Hub archive on metis: append-only, outside both blast radii

Metis (always-on, home infrastructure, Linux) carries the durability role, expressed as a module flag (F7 — moving the hub is flipping the flag on another host and carrying the archive directory across). A systemd timer runs a daily rsync of `~/.claude/projects/` into `~/claude-archive/claude-projects/` with no `--delete` ever, `--backup --backup-dir` into a dated sibling so a corrupted or truncated live file can never silently overwrite the only archived good copy, and Syncthing's transient files (`.syncthing.*.tmp`) excluded. The archive lives outside `~/.claude` (the reaper cannot see it) and outside the synced folder (no propagated deletion can reach it): structurally immune per F4, and Claude-Code-agnostic — it survives format changes, version bumps, and the mesh itself being decommissioned. Known edges, accepted: the archive's capture lag is ≤24h (a session lost or corrupted before its first daily pass is never captured — with a 30-day reap window the realistic exposure is conflicts/fat-fingers in the first day, not the reaper); and a laptop kept offline past 30 days reaps its own unsynced sessions before they ever reach the mesh, which no hub-side mechanism can prevent.

This deliberately replaces the obvious tool, Syncthing's staggered file versioning, for the durability role. Versioning has a structural blind spot here: it only captures changes arriving *from remote devices*. Metis runs `claude` itself, so any transcript whose 30-day sweep fires first on metis is unlinked locally — invisible to metis's own versioning — and the deletion then propagates outward. Which node sweeps first is a startup-timing lottery, so versioning-as-archive would silently miss an unpredictable subset. The rsync archive captures everything regardless of where the sweep fires, with human-readable filenames instead of `.stversions` timestamp-suffixed ones. Recovery runbook: copy the transcript back into `~/.claude/projects/<project>/`, `touch` it (the sweep is mtime-keyed — a restored file with a stale mtime is re-reaped at the next housekeeping pass), and let the mesh fan it out.

### Memory roams too — the CLAUDE.md stance is rewritten, not quietly broken

Agent memory lives at `projects/<encoded-cwd>/memory/` — inside the synced folder — and the operator chose to sync it rather than ignore-pattern it out. Verified against the pinned binary: the retention sweep whitelists file extensions (`.jsonl`, `.cast`, `.ccr-tip.json`) and never touches `memory/*.md`, so memory is durable in the mesh without archive involvement. Consequence: memory becomes fleet-shared and eventually consistent, and the CLAUDE.md "per-host and never synced" memory stance is rewritten in the reconcile step of the landing change — the new stance being that memory roams as an eventually-consistent scratchpad, while git remains the canonical home for anything durable (decisions, conventions, gotchas), because the mesh syncs facts but does not review them. Concurrent memory writes to the same file on two hosts between syncs produce a `.sync-conflict` copy — data preserved, manual pick, expected to be rare.

### Version alignment

The transcript format is documented as internal and unstable. The whole fleet installs `claude-code` from the same `flake.lock`, so version skew is bounded by the gap between per-host switches after a lock bump — an accepted, small window. No separate pin mechanism is needed; there is no non-Nix node.

### Operational protocol (runbook, `docs/claude-session-sync.md`)

One machine per session between syncs: new sessions anywhere are always safe (UUIDs never collide); alternating `--resume` on the *same* transcript on two nodes before sync converges creates a genuine conflict — Syncthing preserves both, manual pick. Let the mesh settle before resuming a just-transferred session. The runbook covers: new-node onboarding (bootstrap → record device ID → re-switch), the one-time Linux `-home-dbf-*` migration, conflict resolution, archive recovery (including the `touch`), and hub migration.

### Verification plan (acceptance criteria)

- **A1** — Session started in `/Users/dbf/code/<repo>` on saturn appears in `claude --resume` on metis (launched from `/Users/dbf/code/<repo>` via the bind mount) after sync, and continues with full history.
- **A2** — Encoded project directory name for a fixture path is byte-identical on a macOS and a Linux node.
- **A3** — Reaper round-trip: a fixture transcript backdated 60 days is swept on next housekeeping pass (policy live), and its pre-sweep content is present in the metis archive.
- **A4** — Divergent edits to one transcript on two disconnected nodes yield a `.sync-conflict` copy after reconnect — no silent loss.
- **A5** — A transcript deleted mesh-wide is restored from the archive, re-synced, and resumable.
- **A6** — Audit on every node: the synced folder contains nothing outside `projects/`, and no credential files exist within it; the eval check from §Mesh is green.
- **A7** — All four participating hosts converge from a clean switch; the only imperative steps are the documented one-time device-ID bootstraps and the Linux cutover migration.

## De-risk evidence

Verified before building, against the pinned toolchain (claude-code 2.1.191 from the repo's nixpkgs pin, on metis, 2026-07-03):

- **Storage layout and path keying** — confirmed against current upstream docs (code.claude.com/docs/en/sessions): JSONL at `~/.claude/projects/<encoded-cwd>/<session-uuid>.jsonl`, encoded-cwd = absolute path with non-alphanumerics → `-`; picker scoped to cwd, Ctrl+W widens to worktrees, Ctrl+A to all projects; format explicitly documented as internal and version-unstable.
- **Sweep is mtime-keyed** — confirmed by reading the shipped binary's housekeeping code (JS source embedded in the bun-compiled executable at the store path): the delete predicate is literally `stat(path).mtime < cutoff → unlink`, no logical-timestamp parsing. A fixture with a 60-day-old mtime but current-dated JSONL content is sweep-eligible. Implication baked into the design: Syncthing preserves mtimes (fine under uniform policy), and archive restores must `touch`.
- **Sweep scope** — same source read: only `*.jsonl`, `*.cast`, `*.ccr-tip.json` files (plus per-session subdirectories and `subagents/workflows/remote-agents` dirs for sessions with no live transcript) are swept. `memory/*.md` is structurally exempt.
- **Sweep trigger** — same source read, explaining an initially-confusing empirical result: housekeeping schedules 5s after interactive startup but defers in 10-minute steps while there was user input in the last 60s, gated by a 24h-mtime sentinel (`~/.claude/.last-cleanup`); `-p` mode exits before the unref'd timer fires. Short-lived test sessions therefore never sweep.
- **Empirical sweep round-trip** — run on metis, 2026-07-03, via a 12-minute idle interactive session (the trigger conditions above): a fixture transcript with 60-day-old mtime but current-dated JSONL content was deleted (mtime-keying confirmed end-to-end), a 10-day fixture survived (threshold confirmed), and a 60-day-old `memory/old-fact.md` survived (memory exemption confirmed), with the sentinel rewritten at sweep time.
- **`cleanupPeriodDays` semantics** — settings schema extracted from the pinned binary: `int().positive()`, described "default: 30. Minimum 1. Use a large value for long retention; use `--no-session-persistence` to disable transcript writes entirely". The historical `0` footgun (#23710, silently disabled persistence) is now schema-rejected; the sweep code also special-cases `0` as skip. Design uses explicit `30`, never `0`.
- **Symlinked settings are broken upstream** (#3575) — and moot here: the repo's ADR-024 jq-merge activation pattern already exists at `home/shared/agent-clis.nix` and is the deployment vehicle for the `cleanupPeriodDays` key.
- **Symlink path unification is dead** — Claude Code calls `fs.realpathSync()` at startup (#39594, #44456), so only a real VFS mount survives; bind-mount syntax and semantics verified against the pinned nixpkgs (`fileSystems` with `options = [ "bind" ]`; mount point must be created, tmpfiles rule).
- **Nix module surfaces** — verified at the pinned nixpkgs/nix-darwin/home-manager revs: home-manager `services.syncthing` runs on Darwin (no platform assertion on the core service) with full declarative devices/folders/versioning/options and `overrideDevices`/`overrideFolders`; `relaysEnabled` and announce toggles exist; `ignorePatterns` is declarative on the NixOS module only.
- **No native alternative** — upstream feature requests #31992/#44063/#52052 all open as of 2026-07; Remote Control requires a live origin process; teleport is web→local one-way. The design is not obsolete on arrival.

Still unverified, carried as implementation-stage checks: Syncthing's behaviour on mid-append JSONL under the fswatcher (expected: transient stale copy, converges next scan — protocol-level, low risk); how the resume picker treats `foo.sync-conflict-<stamp>.jsonl` files (they are sweep-eligible `*.jsonl`, but they may clutter or confuse the picker as pseudo-sessions); the home-manager module's REST-API-based settings application on Darwin; launchd user-agent behaviour on the Darwin hosts when no GUI session is active; metis disk-encryption status (bears on the plaintext-at-rest posture, not on the mechanism); end-to-end A1 itself.

## Drawbacks

- **It inverts two documented stances.** ADR-034 records deliberate non-adoption of backup tooling — and this design's hub archive *is* backup tooling, landing exactly on ADR-034's own migration trigger ("irreplaceable local-only data accumulates"); the wiki design note separately rejected Syncthing as "new machinery"; CLAUDE.md declares agent *memory* per-host-never-synced. This design must carry an ADR that owns these reversals explicitly. The wiki-note rejection reasons (live `.git` corruption) genuinely do not apply to append-only JSONL — but "the reasons don't transfer" must be argued in the ADR, not assumed.
- **The archive is single-disk.** ADR-034 prescribes *off-host* backup plus a tested restore drill when its trigger fires; this design instead creates a data class that exists, past 30 days, solely on metis's single disk. A5 is the restore drill; single-disk durability is operator-accepted for now (inheriting ADR-034's disk-loss acceptance), with an off-host copy explicitly logged as a fast-follow from implementation (§Future possibilities) — the amending ADR must record both halves.
- **mercury's exclusion has a cost.** Work sessions on mercury do not roam and die unarchived at 30 days — the design's primary benefit stops at the work boundary. Deliberate: joining would put personal transcripts at rest on a work AWS box (already on the personal tailnet — `docs/runbooks/headless-bootstrap.md`) and fan work-context transcripts to every personal machine; the operator chose the clean boundary.
- **More plaintext in more places.** Four disks plus an archive now hold every transcript. FileVault covers the Macs; metis/nixos-vm encryption status must be confirmed rather than assumed.
- **A daemon on every participating host.** Syncthing is new attack/maintenance surface on four hosts (localhost-bound GUI, but still a listening service and a package to keep patched) in a repo that has deliberately avoided exactly this.
- **Coupled to unstable internals.** Layout, sweep behaviour, and settings semantics are verified against 2.1.191 only; any release can move them. Mitigated by uniform fleet versioning and the format-agnostic archive, not eliminated.
- **The human protocol is real.** Same-session alternation across machines without letting sync settle produces conflicts the operator must resolve by hand. Rare, preserved-not-lost, but nonzero friction.

## Cost

Standing price once chosen: unbounded (slow, KB–MB/session) archive growth on metis with no pruning policy by design; a one-time device-ID bootstrap per future host; and the `/Users` namespace oddity on every participating Linux host, present and future.

## Rationale & alternatives

- **Syncthing mesh vs rsync/unison**: the fair comparison is not pairwise-O(N²) but a star over the always-on hub this design has anyway. The star still loses on the real grounds: no live propagation (roaming waits for a cron tick), no conflict detection (last-writer-clobbers, worst on a mid-append transcript — exactly the window Syncthing's scan/atomic-rename model handles), and a hub outage stalls all roaming instead of degrading to peer-to-peer. Meets F1/F2 worse at similar machinery.
- **Bind mounts vs symlinks vs `users.users.dbf.home = "/Users/dbf"`**: symlinks are refuted by `realpathSync` (verified, above). Moving the actual Linux home to `/Users/dbf` would unify natively but has fleet-wide blast radius — every path assumption in modules, backups, and muscle memory — for the same result two fstab-equivalent lines deliver. Bind mounts meet F5 at minimum blast radius. A rename/remap sync hook (rewriting encoded dirs in flight) races the sync engine, needs per-pair mapping tables, and mutates mtimes — disqualified by F4's mtime sensitivity.
- **rsync archive vs Syncthing staggered versioning for durability**: versioning's remote-changes-only semantics miss every sweep that fires hub-first (structural, not tunable), and restores from `.stversions` fight timestamp-suffixed names and pruning intervals. The archive job is smaller, Claude-Code-agnostic, and covers all deletion paths. Versioning is dropped entirely rather than kept as a second mechanism — `.sync-conflict` copies plus the archive cover the loss modes; a third overlapping mechanism is enforcement out of proportion (ADR-032).
- **30-day pinned window vs 3650-day effective-keep-forever**: operator's call — 30 days is right in practice for the live picker, keeps laptops lean on sensitive plaintext, and delegates long-term storage to the archive, which solves it better than a decade of live files would (the picker stays usable; the archive is greppable and format-independent).
- **Git as the medium**: excluded by requirement (F2's no-hosted-repo), and wrong anyway — append-heavy JSONL merges noisily, and transcripts adjacent to credentials do not belong in anything with a push remote.
- **Whole-`~/.claude` sync** (community claude-sync tools): syncs credentials and OAuth state between machines — disqualified on its face, and still doesn't solve path keying.
- **Do nothing / wait for native**: the status quo cost is paid weekly (context rebuilt by hand on every machine switch), and the feature requests carry no commitment. The design keeps decommissioning cheap instead (§Future possibilities).

## Prior art

Upstream feature requests #31992/#44063/#52052 document the demand and the absence. Community whole-dir sync tools exist and demonstrate the credential-leak anti-pattern this design scopes around. In-repo: the wiki design note chose git-sync over Syncthing for a git-repo payload (right call there, different payload here); ADR-034 records the no-backup posture this design carves into; the just-landed declarative fleet-SSH work (#517) established the pin-public-keys-in-repo pattern that the Syncthing device graph mirrors; ADR-024 established the settings.json merge pattern this design reuses.

## Unresolved questions

- Resolved during review (operator, 2026-07-03): mercury is excluded from the mesh — the work-boundary drawback was read back and the clean boundary chosen; single-disk archive durability is accepted, with the off-host copy logged as a fast-follow (§Future possibilities).
- During implementation: the unverified items listed at the end of §De-risk evidence; the darwin `.stignore` question (live with `.DS_Store` mac↔mac, or add an activation-copied `.stignore`); whether the Darwin hosts need auto-login or a LaunchDaemon variant for logged-out operation.
- Out of scope: syncing anything outside `projects/` (plans, todos, shell snapshots — each would need its own case); multi-identity folder topology (identity isolation was declined — recorded in `docs/identities.md`, #137 — and is not resurrected here).

## Future possibilities

**Fast-follow from implementation (operator-logged):** an off-host copy of the metis archive — e.g. a periodic pull to neptune — closing the single-disk gap that ADR-034's trigger clause flags for irreplaceable local-only data.

If Anthropic ships native cross-machine session sync, the decommission path is: drop the mesh folder and bind mounts on every participating host (a switch each), keep the metis archive as the historical record, and keep the runbook's recovery section pointed at it. The hub flag makes archive-role migration a flip-and-copy. If Twingate replaces Tailscale (`docs/runbooks/headless-bootstrap.md` records the interim posture), only device addresses change. The mesh pattern (narrow folder, pinned device graph, eval-checked scope) is reusable for any future genuinely-bidirectional payload, though none is currently wanted.
