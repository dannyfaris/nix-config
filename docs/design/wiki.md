# Wiki — a git-synced, LLM-maintained memory layer at `~/wiki`

**Status:** Proposed — design note (`docs/design/`). Build-state: not built. [#506](https://github.com/dannyfaris/nix-config/issues/506). Relates to [docs/desktop/obsidian.md](../desktop/obsidian.md) (viewer selection) and [ADR-034](../decisions/ADR-034-no-host-data-backup.md) (recovery posture); no ADR yet.

## Summary

Establish `~/wiki` — a git repository at the same `$HOME`-relative path on every participating host (neptune, metis, nixos-vm; mercury deliberately excluded) — as the memory layer of the personal OS: an LLM-maintained knowledge base in the [karpathy LLM-wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) shape, viewed and edited through Obsidian, synchronised in the background by home-manager's `services.git-sync` with no user intervention. Nix declares the *infrastructure* (Obsidian the app, the sync service); git carries the *vault* (content **and** its `.obsidian/` configuration, minus volatile state); the wiki's own schema file governs *content conventions*. Mobile access (Obsidian on iOS) is a first-slice requirement.

## Motivation

Durable personal knowledge currently has no home. Claude Code's file-based agent memory is per-host and never synced (CLAUDE.md §"Agent memory lives in git"); notes, reference material, and accumulated context live nowhere structured; and there is no substrate an LLM can both read *and maintain* across the fleet. The personal OS needs a memory layer: a persistent, compounding artifact that agents update as sources are ingested and that the operator can browse, search, and edit as a first-class knowledge base.

The LLM-wiki model supplies the content architecture — raw sources (immutable) → wiki pages (LLM-maintained markdown) → a schema file that makes the LLM "a disciplined wiki maintainer rather than a generic chatbot" — and assumes the vault is a git repo. What it does not supply, and what this note designs, is the *infrastructure*: where the vault lives, what it is called, how it synchronises across hosts and a phone, and where nix's declarative authority stops.

Forces any design must satisfy:

1. **Multi-host, multi-writer reach.** The wiki's writers include LLM tooling operating outside Obsidian, on hosts where the GUI may not even run (nixos-vm). One `$HOME`-relative path must hold on both Darwin and NixOS so tooling references one location fleet-wide.
2. **Zero-intervention, never-destructive sync.** Synchronisation runs in the background without the operator's involvement, and on anything non-trivial it must halt safely rather than guess — a memory layer that silently loses or mangles knowledge is worse than none.
3. **Mobile is a requirement.** The vault must be readable and writable from Obsidian on iOS in the first slice, which constrains both the sync design and where vault configuration lives (a phone cannot receive nix-managed files).
4. **Three authorities must not fight.** Nix (machine/app configuration), git (content transport), and the wiki's schema file (content conventions) each want a say inside the vault directory; their boundaries must be drawn deliberately or they will overwrite each other.
5. **The repo's standing posture holds.** Vault *contents* are outside nix-config's PRD scope (§2.2); recovery remains reprovision-from-flake + off-host durable data (ADR-034); the work/personal identity boundary holds (mercury is work-only); macOS TCC-protected directories are avoided so agents and CLIs can read the vault without permission friction; explicit > implicit and whitelist > blanket throughout.
6. **Obsidian is the viewer, not the owner.** The artifact is portable markdown; the design must not couple the vault's identity, location, or conventions to the current viewer.

## Design

**Location and name: `~/wiki`.** Home root, not `~/personal/` — the wiki is infrastructure, a peer of `~/nix-config` (the declarative machine layer beside the declarative memory layer), not one project among many. The same relative path holds on Darwin (`/Users/dbf/wiki`) and NixOS (`/home/dbf/wiki`), mirroring the `flakePath` precedent (ADR-019). The name follows the taxonomy's most-communicative-term rule: "wiki" names the role honestly; it is not tool-named (Obsidian is the viewer, force 6) and it does not borrow the celestial vocabulary, which ADR-038 reserves for hosts. Sitting outside `~/Documents`/`~/Desktop` keeps it clear of macOS TCC prompts (force 5), and on the dual-identity hosts a repo at home root falls through to the personal git identity — correct, since mercury (the only forced-work-identity host) is excluded.

**Host scope.** neptune, metis, and nixos-vm carry the vault and sync service; **mercury is deliberately excluded** — personal memory does not belong on the work-only host (force 5). The Obsidian GUI lands on neptune (existing Homebrew cask) and metis (`pkgs.obsidian`, a deliberate `allowUnfreePredicate` whitelist addition); nixos-vm gets vault + sync only (desktop modules are not installed there — UTM cannot render Wayland compositors).

**Sync: home-manager `services.git-sync`, fleet-wide.** One declarative mechanism on both platforms, verified in the pinned home-manager (see De-risk evidence), with a real platform asymmetry that peer review surfaced and the pinned sources confirm. On Darwin: a launchd agent invoking `git-sync` bare, on an interval (module default 500 s — kept for the first slice; a default, deliberately called one; tuning waits for P3 and real cross-device usage) plus `WatchPaths` on the vault — but `WatchPaths` is a *non-recursive* watch on the vault root, so edits nested under `wiki/` or `raw/` do not fire it; for real content Darwin is effectively interval-paced. On NixOS: a systemd user service running the `git-sync-on-inotify` wrapper — genuinely change-triggered (recursive `inotifywait`, `.git`-excluded) with the same interval as fallback. The underlying `git-sync` script autocommits local changes, fetches, then reconciles by state: ahead → push; behind → `merge --ff-only`; diverged → rebase, and on a conflicted rebase it **halts and waits for manual resolution** — it never force-pushes or auto-resolves (force 2). Two desktops firing simultaneously resolve through the same state machine: the losing push fails and reconciles on that host's next cycle.

**The whitelist is the unit declaration, not the branch flags.** git-sync's per-branch opt-in (`branch.main.sync` / `branch.main.syncNewFiles`) is honoured only where the script is invoked bare — Darwin. The Linux wrapper passes `-n -s` on every invocation, force-overriding both flags, so on NixOS the unit syncs its declared repo unconditionally, new files included. This is accepted, not mitigated: the load-bearing whitelist is that `services.git-sync` creates units *only for explicitly declared repositories* — the daemon touches `~/wiki` and nothing else; the branch flags are a second gate where the platform provides one, not the design's safety property. Bootstrap is therefore per-platform: **neptune** — clone `~/wiki` manually (the module's `uri` clone-if-missing is unsupported on Darwin), set the two branch flags, activate; **NixOS hosts** — declare `uri` (mandatory there: the unit references it) and the wrapper clones on first start. The remote is a **private GitHub repository, `dannyfaris/wiki`** (confirmed). Pushes authenticate over HTTPS via the fleet's existing `gh` credential-helper wiring (ADR-009) — no SSH keys involved.

**Nix seam and placement.** One standalone home module, `home/shared/wiki.nix` (most-communicative-term: it configures the wiki's machine plumbing, currently the `services.git-sync` repository entry), imported by neptune, metis, and nixos-vm. Mercury's exclusion is enforced by the composition model itself — a host has only what it imports (ADR-027); no further mechanism is warranted. The Obsidian GUI rides existing seams: neptune's Homebrew cask stands; the metis slice adds `pkgs.obsidian` with its `allowUnfreePredicate` whitelist entry and reconciles docs/desktop/obsidian.md (Darwin-scoped today) in the same change.

**Authority partition (force 4).** Three authorities, three territories:

- **Nix** owns the machine side: the Obsidian application, the git-sync service, and nothing inside the vault. It does *not* adopt `programs.obsidian` (see Rationale — the module's mechanism conflicts with both git transport and the recorded cask update stance).
- **Git** carries the vault: content (`wiki/`, `raw/`, `index.md`, `log.md`, the schema `CLAUDE.md`) **and** `.obsidian/` configuration — settings, plugin list, plugin data — minus volatile state. This is what makes mobile parity work: the phone's Obsidian receives the same settings and community plugins through the same git transport (force 3). A vault-level `.gitignore` excludes the churn: `.obsidian/workspace.json`, `.obsidian/workspace-mobile.json`, `.trash/`, `.DS_Store`.
- **The wiki's schema file** (`~/wiki/CLAUDE.md`) owns content conventions — page types, frontmatter, naming, ingest/query/lint workflows — per the LLM-wiki model. Content conventions are explicitly *not* designed in this note and *not* recorded in nix-config: they live in the vault, versioned with the content they govern, co-evolved by operator and LLM (force 5's scope boundary, kept).

**Vault shape (recorded here only as the interface the infrastructure serves).** Per the LLM-wiki model: `raw/` immutable sources, `wiki/` LLM-maintained pages, `index.md` catalog, `log.md` append-only record, schema at root; Obsidian's attachment folder pinned to a fixed directory (`raw/assets/`) — an `.obsidian/app.json` setting, carried in git like the rest.

**Mobile (force 3).** The phone joins as another git client, not another sync system. **Working Copy is the confirmed top contender**, to be tried first (10-day pro trial covers the verification before the one-time unlock): a full iOS git client that folder-syncs the repo into Obsidian's container, automated via Shortcuts on Obsidian open (pull) and close (commit + push). It wins on robustness — a real git implementation where the vault's `raw/` asset growth is the documented weakness of the alternative — and on architecture: the phone's sync machinery stays *outside* the vault, exactly as git-sync does on desktop. The fallback is **obsidian-git on mobile** (in-app, isomorphic-git, HTTPS + token): free and one-app, but with documented mobile limits (no SSH, memory constraints, minutes-long change detection on larger vaults) and a design wrinkle — plugin enablement travels in the git-carried `.obsidian/`, which would load a second (dormant) committer on the desktops. On-device verification of Working Copy is a first-slice task; nothing in the desktop design depends on the outcome, precisely because `.obsidian/` travels in git.

**One committer per device class.** On desktops, git-sync is the sole automatic committer — obsidian-git is *not* installed on desktop (two auto-committers racing one repo buys only conflicts). Agents may still make semantic commits at meaningful boundaries; git-sync pushes them. On mobile, whichever client wins is that device's committer.

How the design meets the forces: one path everywhere (1); a verified halt-on-conflict sync whose reach is whitelisted by unit declaration (2); mobile parity via git-carried vault config (3); a three-way authority partition with nix stopping at the vault boundary (4); mercury excluded, TCC avoided, PRD scope kept, unfree whitelisted explicitly (5); a role-named, viewer-agnostic artifact (6).

## De-risk evidence

Verified 2026-07-02 against the flake pins on neptune:

- **`services.git-sync` exists and is cross-platform in the pinned home-manager** (`/nix/store/ya4hl1n0yp7pm490aviz59fcz0nwlrda-source/modules/services/git-sync.nix`): a launchd agent on Darwin (`StartInterval` at the module default + `WatchPaths` on the repo root — non-recursive, per launchd semantics) and a systemd user service on Linux (`ExecStart = git-sync-on-inotify`, `Restart = on-abort`, no flags on Darwin's bare `git-sync` invocation, no stdout/stderr paths declared on the launchd agent). Caveats found: the `uri` clone-if-missing option is documented as unsupported on Darwin (neptune clones manually) and is effectively mandatory on Linux (the unit's environment references it).
- **`git-sync`'s safety model, read in the pinned nixpkgs script** (`git-sync-0-unstable-2025-06-26`, `bin/.git-sync-wrapped`): fetch, then ahead → push; behind → `merge --ff --ff-only`; diverged → `git rebase`, and on failure "Resolve them and finish the rebase before repeating git-sync" — a halt, not a guess. Sync and new-file commit are opt-in per branch (`branch.<name>.sync`, `branch.<name>.syncNewFiles`, with global `git-sync.*` fallbacks) — **but honoured on Darwin only**: peer review caught, and the wrapper source confirms, that `git-sync-on-inotify` passes `-n -s` on every invocation (startup, timeout, and change paths), force-overriding both flags on Linux. Its clone path sets only `branch.<name>.pushRemote`. The wrapper's loop also ignores `git-sync`'s exit codes — a conflicted halt leaves the unit alive and re-refusing each cycle; `Restart = on-abort` matters only if the wrapper itself dies. The design's whitelist framing and P7's expectations are stated accordingly.
- **`programs.obsidian` exists in the pinned home-manager and was read in full** (`modules/programs/obsidian.nix`): it can declare per-vault `app.json`, `appearance.json`, core/community plugins (as nix packages, with settings), hotkeys, themes, and CSS snippets. Two conflicts with this design were identified from the source: (a) it manages vault settings as **read-only nix-store symlinks inside `<vault>/.obsidian/`** — inside the git-synced tree, where they would be committed as store-path symlinks (broken on other hosts) and would fight Obsidian's own runtime writes; (b) its activation unconditionally writes `updateDisabled = true` into `obsidian.json`, contradicting the recorded neptune stance that the cask updates via Obsidian's built-in updater (docs/desktop/obsidian.md). Hence: not adopted.
- **The LLM-wiki requirements** were read from the karpathy gist (2026-07-02): vault-as-git-repo, `raw/` + `wiki/` + `index.md` + `log.md` + schema-at-root, wikilinks + YAML frontmatter, attachment folder pinned to a fixed directory.
- **Git auth wiring** verified in the deployed home-managed git config (`~/.config/git/config` on neptune): `credential."https://github.com".helper` invokes `gh auth git-credential` by absolute nix-store path — so git-sync's pushes have no PATH dependence on the launchd/systemd environment; the remaining question is token reachability from a background context (probe P4 below).

Still unverified — the on-host runtime probes for the build slices (set ≠ enforced, CLAUDE.md §Conventions). Each probe records its result here as the slice lands:

- **P1 — unit liveness.** The launchd agent is loaded in neptune's gui domain (`launchctl print gui/$UID/…git-sync-wiki`) and the systemd user unit is active on metis (`systemctl --user status git-sync-wiki`).
- **P2 — local-edit propagation (watch trigger).** Edit a file in `~/wiki`; expect an autocommit and push observed at the remote (`git ls-remote`), not just locally. On neptune the edit must be a *nested* file (e.g. under `wiki/`) — `WatchPaths` is non-recursive, so a root-level test could pass while real content rides the interval only; the probe measures which trigger actually fires and the resulting latency. On metis (recursive inotify) expect seconds.
- **P3 — remote-edit pull (interval trigger).** Commit at the remote with *no* local filesystem activity; expect the local vault to fast-forward within one interval. This is the probe that defines cross-device latency — a phone edit reaches a desktop only via this path, since `WatchPaths`/inotify cannot see remote changes.
- **P4 — non-interactive auth.** A push from the launchd context on neptune: the credential helper is an absolute store path (verified above), but `gh` must reach its token from a background agent's bare environment (macOS keychain access outside an interactive shell). Same probe on metis under systemd (`gh` token storage headless).
- **P5 — new-file pickup.** A brand-new file commits: on neptune this confirms the branch-flag bootstrap took (the flags are load-bearing there); on metis it should happen regardless (`-n -s`), confirming the platform asymmetry reads as designed.
- **P6 — ignore boundary.** `.obsidian/workspace.json` churn does *not* commit. The volatile set is expected to widen (e.g. `graph.json`, per-device plugin `data.json`) — this probe seeds the real ignore list rather than confirming a final one.
- **P7 — conflict halt and aftermath.** Manufacture a same-line conflict between two hosts; confirm git-sync halts without destructive action and the vault stays editable, then observe the aftermath the sources predict — on both platforms the unit stays alive and re-refuses each cycle (launchd re-fires; the Linux wrapper's loop ignores exit codes) — and where the failure is visible (`journalctl --user` vs macOS `log show`; the launchd agent declares no stdout/stderr paths). This probe's findings feed the halted-sync-observability question.
- **P8 — Obsidian mid-write interplay (light).** Edit in Obsidian while a sync fires; confirm per-file atomic writes and autocommits coexist without corruption. Low expected risk; observe once.
- **P9 — Working Copy on-device.** Clone, folder-sync into Obsidian's container, open/close Shortcuts automations, round-trip against a desktop — within the pro trial window.
- **`pkgs.obsidian` on metis under niri** (Wayland/Electron behaviour) — at the metis slice.

## Drawbacks

- **Personal knowledge auto-pushes to a remote.** A background service committing and pushing everything in `~/wiki` to GitHub widens the privacy surface — anything dropped in the vault leaves the machine within minutes, and history makes removal effectively permanent. The vault must be treated as remote-bound by definition; secrets never enter it.
- **A halted sync is silent.** git-sync's safe-halt (its best property) has a flip side: after a conflicted rebase the vault quietly stops syncing until someone notices. No monitoring is designed here; divergence between hosts can accumulate unnoticed.
- **First user-data infrastructure in nix-config.** This config has so far declared *configuration only*; a fleet-wide service tending a personal-data directory is a precedent, however carefully bounded. The scope boundary (nix stops at the vault wall) is the mitigation, but the precedent stands.
- **Commit history as noise.** Autocommits are timestamped ("changes from <host> on <date>"); the wiki's git log is a sync journal, not a curated history. Acceptable for a memory layer (the LLM-wiki model says a nice history "is not crucial, but having one at all is") — but it forecloses history-as-documentation.

## Rationale & alternatives

**Location.** `~/personal/wiki` was the serious alternative (identity wiring for free, tidy home root) — rejected because the wiki is not one project among many but a standing layer of the personal OS; the `~/nix-config` home-root precedent is the honest parallel, and identity resolves identically anyway (personal is the fallback everywhere the vault exists). `~/Documents/...` — the conventional macOS Obsidian placement — is rejected on TCC friction (every CLI touching the vault would trip permission prompts) and platform asymmetry. iCloud Drive placements fail forces 1 and 3 (no NixOS presence; git-in-iCloud is a corruption pattern).

**Name.** "wiki" over: `obsidian-vault`/`vault` (tool-named — couples the artifact to the current viewer; the taxonomy admits tool names only when the tool *is* the role); `memory` (honest but collides with the agent-memory vocabulary already in CLAUDE.md, and the artifact is broader — the gist's own term is wiki); `mnemosyne` or other celestial/mythological names (ADR-038 reserves the celestial scheme for hosts; a vault is not a machine, and borrowing would blur a clean taxonomy).

**Sync mechanism.** Against the field: **obsidian-git on desktop** syncs only while the app runs — disqualified by agent writers and GUI-less hosts (force 1). **gitwatch** is a committer, not a synchroniser — no reconciliation story (force 2). **git-annex assistant** fights Obsidian's file model and over-tools a markdown repo. **Syncthing/iCloud over the repo** syncs a live `.git` directory (corruption pattern) and discards inspectable history; Syncthing would also be new machinery in a repo that has deliberately avoided it (ADR-034 posture). **Obsidian Sync** is opaque to CLI agents, subscription-bound, and unusable on GUI-less hosts. `services.git-sync` is the only candidate that is declarative, cross-platform in the existing pins, and safe-halting — scoped to explicitly declared repositories, with the per-branch opt-in as a Darwin-only second gate.

**`.obsidian/` ownership.** Three options: (A) nix owns it via `programs.obsidian`, git ignores it; (B) git carries it, nix stays out; (C) as B but ignoring volatile workspace state. **C wins.** A fails three ways, all verified from the module source: store-symlinks inside the synced tree, Obsidian's runtime writes against read-only targets, and the forced `updateDisabled = true` contradicting the recorded cask update stance — and it fails mobile (force 3), since a phone cannot receive nix-managed plugins. B without the ignore list commits `workspace.json` churn — the main realistic conflict source — into every sync. C gives mobile parity, keeps nix's scope boundary clean (force 5), and minimises conflict surface.

**Doing nothing** leaves personal knowledge scattered and agent memory per-host — the problem statement persists by default.

## Prior art

- **The karpathy LLM-wiki gist** — the content architecture this infrastructure serves; its assumptions (vault-as-git-repo, schema-at-root, fixed attachment path) are inputs to this design rather than restated conventions.
- **This repo's own stances:** CLAUDE.md §"Agent memory lives in git, not local state" is the same conclusion at smaller scale — durable knowledge belongs in a synced git repo, not per-host state. ADR-034 ("durable data has homes off the host") supplies the recovery posture: the wiki's backup *is* its remote. docs/desktop/obsidian.md selected the viewer and recorded that nix does not manage vault contents; this note keeps that line and draws it precisely.
- **The Obsidian community's git-sync ecosystem** (obsidian-git, Working Copy folder-sync on iOS) is well-trodden prior art for the mobile leg; the desktop leg deliberately departs from the community default (in-app plugin) for the reasons above.

## Unresolved questions

- **Mobile client** — Working Copy confirmed as top contender, tried first (P9); obsidian-git mobile is the fallback if on-device verification disappoints.
- **Halted-sync observability** — out of scope for the first slice, but the silent-halt drawback wants at least a lightweight answer eventually (a prompt/statusline surface consistent with the existing visual vocabulary, or a periodic check). Deliberately not designed here.
- ~~Slice order~~ — decided: **neptune first** (Obsidian and the riskiest probe variant, P4-on-keychain, both live there — fail fast where failure is most likely), then metis; nixos-vm follows whenever convenient, adding no new probe classes.

## Future possibilities

- **`programs.obsidian`, revisited** — if the module gains a mode that composes with git-carried vault config (or for settings genuinely per-machine), the verified read of its mechanism is already on file here.
- **Search and MCP** — the gist's optional qmd (hybrid local search, CLI + MCP server) would give agents retrieval over the wiki; a natural later slice, possibly its own selection exercise.
- **hostContext wiring** — if tooling needs the wiki path per-host, `hostContext` is the established seam (ADR-019); not warranted while the path is a fleet-wide constant.
- **Stylix theming of Obsidian** — docs/research/omarchy-theme-switching.md notes obsidian.css templating prior art; could join the theming surface later.
- **Publishing** — the wiki's shape (index, wikilinks, markdown) leaves static publishing open if parts ever warrant sharing.
