# Engineering review — design discipline, code quality, and the issue backlog (2026-07-06)

Status: **point-in-time review artifact, not a decision.** Captured 2026-07-06 — an operator-requested full-repo engineering review by Claude (Fable 5). Scope: elegance and discipline of the design, elegance and maintainability of the code, quality of the core engineering, and implementation-level improvements — across the flake and the GitHub issue backlog. **Deliberate scope exclusion at operator request:** documentation volume, comment size, and process/ceremony weight were actively excluded from consideration (a known issue that would have overwhelmed the rest); absence of commentary on those axes is not endorsement or critique. Method: direct read of the core plumbing and roughly 60% of the Nix by hand; three parallel read-only sweep agents over the remaining modules, home layer, and tooling, with every relayed finding independently re-verified (two sweep findings were rejected on verification and do not appear below); empirical probes (`nix eval` against the capability registry, sed under BSD vs GNU, statix/deadnix runs, byte-level diffs). Reviewed at working tree `315b2ba` (`feat/cycle-ghostty-windows`, then-current). Feeds: future issues — none filed as of capture.

## Verdict

This is a genuinely well-engineered configuration. The review found no correctness bugs in the Nix core, the security posture is implemented as tightly as it is described, and several mechanisms (the stance checks, the bundle-purity linter, the capability registry's test suite) withstood deliberate attempts to find evasions — the bundle-purity regex was probed for malformed-input bypasses and holds, because unparseable input dies at `nix-instantiate --parse` before the shape regex runs. The findings below are almost all of one species: **systems whose guarantees stop one step short of where the design implies they reach**. The repo consistently builds the right instrument and then doesn't quite point it at everything.

## 1. The capability registry trusts its own data shape (highest leverage)

`lib/capabilities.nix` is the strongest artifact in the repo — and its weak edge is that the registry is untyped data validated only by what the emitters happen to match. Confirmed empirically: a capability with `realization = "niri-actoin"` (typo) evaluates to `{ emittedBinds = { }; collisionLint = [ ]; }` — silently dropped from emission *and* invisible to the collision lint. The keybinds-table diff check catches this accidentally for *existing* entries (the row flips to "—" and CI fails against the committed table), but a *new* entry added with a typo'd tag gets its "—" baked in by `just gen-keybinds` and sails through. For a system whose reason to exist is "no silent drift," this is the missing rung: a registry-shape validator (required fields present, `realization` ∈ known set, `chord.tier` ∈ `tiers`, darwin `key` ∈ the verified `asKey` map ∪ letters/digits) run as one more eval check. Cheap, and exactly in-idiom with the lib unit tests already gated in CI.

Three adjacent gaps in the same seam:

- **`aerospace-exec` chords are restated, not looked up.** `home/darwin/aerospace.nix:50-67` re-declares the chord literals (`{tier="hyper"; mods=["Shift"]; key="M"}`) and only shares the *renderer* with the registry. If the registry entry for `maximise-by-isolation` moves to a different chord, aerospace.nix keeps binding the old one, the collision lint sees no collision, and the generated keybinds table now documents a chord that isn't bound. The header comment ("their chords come from the registry… they cannot drift from the caps they realize") overclaims — same renderer ≠ same data. Fix: look the chord up by capability id, and assert at eval that the set of `aerospace-exec` ids in the registry exactly equals the set of hand-authored bodies. That also catches the converse failure (a fifth exec cap added to the registry with no body: reserved, documented, and inert at runtime).
- **Merge-guard asymmetry.** `home/nixos/niri.nix:38-45` merges generated over hand-authored binds through `mergeBinds` with a hard `throwIf` on shadowing; `home/darwin/aerospace.nix:147` merges with bare `//` and relies on the convention that hand binds are also registry-declared. Mirror the throwIf.
- **Error-surface asymmetry.** A bad key token on Linux fails at build (`niri validate`); on Darwin, `asKeyFor`'s `lib.toLower` fallback (`lib/capabilities.nix:149`) defers it to AeroSpace's runtime config parse — which rejects the *whole file*, killing every keybind on the host at the next reload. The key-token whitelist above moves that to CI.

## 2. The mirrored-twin tax

The platform-pair pattern is applied even where the twins are one expression apart, and each pair carries a hand-maintained "mirrors X" contract:

- `modules/{nixos,darwin}/host-context.nix` — 46/42 lines differing in exactly one default value.
- `modules/{nixos,darwin}/home-manager.nix` — ~80 lines each, differing in three values (`homeDirectory`, `zellijCacheDir`, `flakeConfigAttr`).
- `modules/{nixos,darwin}/stylix-palette.nix` — the scheme-selection `let` block (including the polarity-throw logic) duplicated verbatim, with a third variant of the same palette lookup in `lib/scheme-pair.nix`.

The repo already owns the correct instrument for this: plain `lib/` records and functions (`operator.nix`, `scheme-pair.nix`). A constructor in `lib/` taking a small platform record (`{ home, zellijCacheDir, flakeConfigAttr }`) would collapse each pair to a two-line platform file without touching the `shared/`-purity rule — these would be lib functions, not shared modules, so no platform conditional enters `shared/`. Where twins genuinely diverge (firewall, sshd, nix-daemon), the split is earning its keep; these three pairs aren't diverging, they're photocopying.

The same species at the bash layer, which #339 already names precisely: ~180 lines shared between `home/shared/claude-statusline.sh` and `home/shared/cursor-statusline.sh` (git porcelain parsing, `vw()`/`pad2()`, path shortening), the porcelain counter a third time in `home/shared/multiplexer.nix:139-148`, and the glyph byte-tables duplicated while the *colours* are already correctly Nix-generated into `statusline-colours.sh`. The fix is the pattern already built, extended one file further: generate a shared prelude (glyphs, width tables, git-state function, identity→label map) the way the colours are generated. Two additions to #339's scope worth folding in: the identity emails live as literals in **four** places (`home/shared/git-identity-dual.nix:17,25`, `home/shared/git-identity-work.nix:13`, `home/shared/claude-statusline.sh:76-77`) and belong in `lib/operator.nix`, which exists precisely to be this record; and `home/shared/multiplexer.nix:175-190`'s help-screen text hand-mirrors the binds defined 100 lines below it in the same file — same drift class, same file.

## 3. Composition gaps at the host layer

- **neptune and saturn's `extraHomeModules` lists are entry-for-entry identical** — verified by diff: 15 restated paths each, zero divergence. Their entire real difference is two system imports (`sshd.nix`, `power.nix`). The Linux side got `home/nixos/bundles/desktop-env.nix` for exactly this shape; `home/darwin/bundles/` doesn't exist. One `macos-workstation` home bundle would make saturn's file the two-line diff it conceptually is, and turn "add a module to both Macs" from a two-file edit that can half-land into a one-file edit.
- **The hostname is stated three times per host** (the `mkHost` argument, `networking.hostName`, `hostContext.hostName`) with nothing asserting agreement. A copy-paste mismatch is silent and non-obvious: `lib/host-palettes.nix` would theme the host as a *different valid host*, and nixd would complete options against the wrong configuration. Either default `hostContext.hostName` from `config.networking.hostName`, or have `mkHost` inject both from its argument.
- **`parts/checks.nix:133-172` hand-restates the host→system mapping** that `parts/{nixos,darwin}.nix` already define. Deriving the per-host checks by iterating `self.nixosConfigurations`/`darwinConfigurations` (grouping by `hostPlatform.system`) would make "add a host" touch two files instead of four. CI is already clean here — the matrix is per-arch and discovers hosts through `flake.checks`.
- **`lib/display-profiles.nix:23`** — the `active = "2.0"` knob is repo-global but is per-host state (it's metis's panel). Correct today with one desktop host; the moment a second desktop host lands, this belongs on `hostContext`. Worth moving before the shape calcifies.

## 4. Dead machinery kept live

`lib/capabilities.nix:1068-1071` keeps `karabinerHyperRemapKeys` as permanently-empty lists, and `home/darwin/karabiner.nix:114-159` keeps two full manipulator-generating productions (plus the empty-rule filter whose only real justification is guarding those generators) to consume them. The keep-rationale is circular — the attr is kept because karabiner.nix reads it; karabiner.nix reads it because it exists — and ADR-040 declares the retirement permanent. Delete both halves; git history preserves the recipe if Mission-Control remaps ever return. Smaller instances of the same: `lib/theme-tokens.nix:102-105`'s empty `motion` placeholders, and the dead `or [ ]` fallbacks on option-guaranteed values in both `home-manager.nix` files (`hostContext.extraHomeModules or [ ]` — the option's default makes the fallback unreachable and quietly implies the field might be absent).

## 5. Tooling traps (all verified on-box)

- **`justfile:33,92,170` hardcode `/dev/shm/nix-bootstrap-*`** in `gen-host-key`, `bootstrap`, and `bootstrap-clean`, while sibling `setup-sops-identity:232-235` already solved macOS portability with a `uname` case and an explanatory comment. Since the operator's daily machines are now Macs, the *bootstrap path for a new or reinstalled NixOS host* fails exactly when it is needed. The first fix to make from this section.
- **`scripts/test-lint-design-note.sh:141`** uses GNU-only one-line `sed …i text\n` syntax. Under `/usr/bin/sed` it hard-errors ("command i expects \ followed by text"); it works on the fleet's Macs today only because home-manager's gnused shadows BSD sed on PATH. A contributor or context without that environment gets a broken self-test. (GNU sed does build the fixture correctly — the issue is portability, not fixture correctness.)
- **`http://metis:8090/fleet-failures`** is restated across `modules/nixos/unit-failure-notifier.nix:38` and `modules/nixos/ntfy-server.nix:20,23` — receiver host, port, and topic are a three-literal contract maintained by parallel edits.
- Trivial: the six identical Sparkle key blocks in `modules/darwin/homebrew.nix:126-157` are a `lib.genAttrs` over a commented id list.

## 6. The issue tracker as an engineering artifact

The backlog is unusually honest about its own debt — #339 quantifies the statusline duplication to the line, #526's key-rotation sequencing is careful (old key retains decrypt until re-encryption completes), #464 correctly identifies that the 3-arch build matrix is the CI floor pre-commit can't absorb, and #303/#420/#421/#346 track exactly the set-vs-enforced gaps a config like this accumulates. What is *not* tracked is essentially the list above: registry shape-validation and the exec-bind completeness assert, the Darwin home-bundle gap, the hostname triplication, and the `/dev/shm` bootstrap break. None of these were named anywhere the review looked — they'd be new issues, not reprioritizations.

## What holds up under pressure

Worth stating because these were actively probed for weaknesses: the stance layer's Darwin SSH check (comment-filtering, exact-token normalization — the #344 fix is real defense, and its independence from the module's own values is the right call, since deriving both from one source would make the check tautological); `session-type`'s process-ancestry detection with its cache; the `appearance.onChange` typed hook option with per-hook isolation and logging; the empty-but-kept `stylix-targets.nix` whitelist as an explicit surface; the fail-loud polarity/scheme coupling; and the registry's test suite, whose live-registry guard tests (`testLiveRegistryClean`, the merged-namespace test) are exactly the shape unit tests around generated config should have. statix and deadnix both come back clean outside the deliberately-carved-out generated hardware configs.
