# ADR-018: Headless secrets — continue with sops-nix

> **Amended by [ADR-022](./ADR-022-headless-bootstrap-nixos-anywhere.md) on 2026-05-25** for the host SSH key acquisition order only. ADR-022 shifts host-key generation from post-boot harvest (described in this ADR's Implementation section and the original AWS runbook) to operator-side pre-generation injected via `nixos-anywhere --extra-files`. The core decision below (sops-nix, age recipient per host, SSH host key as decryption identity, plaintext blob safe in a public repo) is unchanged and remains authoritative for everything except acquisition order.

**Date**: 2026-05-18
**Status**: Accepted

## Context

PRD §2.2 and §12 deferred the question of how `headless` instances should consume runtime secrets. The PRD's workstation-side answer is 1Password's CLI (`op`) at runtime, with no nix-managed encrypted secrets. The headless answer was left open because the PRD considered headless hosts not to require runtime secrets at the time of writing.

That is no longer quite true. The first concrete headless instance (Mercury, ADR-017) needs at minimum a hashedPasswordFile for the `dbf` user — the same secret the UTM VM already consumes via sops-nix. The choices considered were (a) continue with sops-nix on headless hosts, mirroring the VM's existing pattern; (b) introduce 1Password service-account tokens, where a bootstrap-provisioned token lives on the host and is used by `op` to fetch secrets at runtime; or (c) defer entirely and provision dbf with a temporary password set imperatively, accepting that `users.mutableUsers = false` makes this awkward.

## Decision

Headless hosts continue to use sops-nix, identically to the UTM VM. The host's existing ed25519 host SSH key, generated by cloud-init on first boot, is the decryption identity via the `age.sshKeyPaths` setting in `modules/core/nixos/sops.nix`. Adding a new host means harvesting its post-cloud-init host pubkey, converting it to an age recipient (`ssh-to-age`), adding the recipient to `.sops.yaml`, and re-encrypting via `sops updatekeys`. The procedure is documented in `docs/runbooks/headless-bootstrap.md`.

1Password `op` on headless is not adopted at this point. The PRD's preference for `op` over sops is real but pays off where the secrets are diverse, frequently rotated, or shared with other identities — none of which apply to Mercury today. Re-evaluation when a real headless workload (e.g. a long-running service consuming multiple rotating credentials) appears is the correct trigger.

## Rationale

The decision is the path of least friction. sops-nix is already wired in this repo (PRD §2.2 explicitly mentioned its prior introduction in Tier 2), the encryption key is the host's SSH key which exists by the time of the first rebuild, and the encrypted blob in `secrets/secrets.yaml` is safe to publish even after the repo goes public. Adding Mercury costs one runbook step (the recipient onboarding) and zero new infrastructure.

The 1Password route looked attractive on paper because workstation hosts already use `op` and unifying the two would simplify the mental model. But on a headless host, `op` requires a service-account token to be provisioned out-of-band and stored somewhere on the host — which is itself a secret, and the bootstrap problem doesn't go away, just moves. The token also makes the host's identity in 1Password's audit trail less specific than per-secret sops recipients are in `.sops.yaml`. Neither downside is fatal, but they're real, and there's no concrete value to weigh them against until a headless workload that benefits from runtime secret access exists.

The "defer entirely" option (temporary password) was rejected because `users.mutableUsers = false` means a temporary password set imperatively at first boot evaporates on the next rebuild, locking the user out. The hashedPasswordFile mechanism is the supported declarative path; sops-nix is the supported way to keep that file out of the working tree.

## Consequences

- ✓ Mercury bootstrap reuses the VM's sops-nix wiring unchanged. No new module, no new dependency, no new ADR-level mental model.
- ✓ The encrypted blob is safe in a public repo. The age recipients in `.sops.yaml` are public keys by design.
- ✓ Per-host recipients in `.sops.yaml` give per-host audit granularity — a recipient can be removed (via `sops updatekeys`) if a host is decommissioned, and the decryption is genuinely revoked at the file level.
- ✗ Adding a host requires a recipient-onboarding step that touches `.sops.yaml`, `secrets/secrets.yaml`, and a git commit. This is a one-time-per-host operation but it is genuinely procedural, not zero-effort.
- ✗ The age key is bound to the host's SSH key. Regenerating the host SSH key (rare but possible — e.g. if it's exposed) requires updating `.sops.yaml` and re-encrypting, the same as adding a new host.
- ⚠ Migration trigger: a real headless workload that consumes multiple rotating credentials, or one that needs short-lived access to secrets that don't sit comfortably in a long-lived encrypted blob (e.g. cloud-API tokens), is the point to re-evaluate. At that point, `op` with a service-account token (or HashiCorp Vault, or AWS Secrets Manager) becomes worth the bootstrap step.

## Implementation

Existing `modules/core/nixos/sops.nix` is unchanged. `age.sshKeyPaths = [ "/etc/ssh/ssh_host_ed25519_key" ]` and `defaultSopsFile = ../../../secrets/secrets.yaml` cover both hosts.

Per-host onboarding adds the host's age recipient to `.sops.yaml`'s `keys:` block as an anchor, then includes that anchor in the existing `creation_rules` `key_groups` list. `sops updatekeys secrets/secrets.yaml` re-encrypts the existing blob for the expanded recipient set. The runbook at `docs/runbooks/headless-bootstrap.md` documents the exact sequence; this ADR captures the why.
